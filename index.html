学习路线：先会用，再讲透，最后能排坑

很多人学协程会卡在两个极端：只背 API，遇到异常和取消就崩；一上来啃源码，缺少 Android 使用场景。建议按下面顺序推进。

第一层：把协程当成“可取消的异步任务”

掌握 launch、async/await、withContext、viewModelScope，能把网络、数据库、文件 IO 从主线程挪出去。

第二层：理解 suspend 的真实形态

suspend 不创建线程，不等于异步。它只是允许函数在挂起点保存现场、返回、稍后恢复。

第三层：用 Job 树解释生命周期

父取消子、子异常影响父、兄弟协程如何被级联取消，这些都由结构化并发和 Job 状态机控制。

第四层：把 Flow 当成数据管道

冷流负责声明数据生产逻辑，热流负责状态共享。理解 flowOn、catch、stateIn 才能写稳定 UI。

第五层：面向 Android 架构落地

ViewModel 暴露状态，Repository 暴露 suspend/Flow，UI 按生命周期收集，测试里控制 Dispatcher 和虚拟时间。

为什么需要协程：它解决的不是“少写线程”，而是异步控制权

学习协程之前，先不要急着背 launch、async、withContext。更重要的问题是：Android 开发为什么会需要一种新的异步模型？答案不是“线程太重”这么简单，而是传统异步代码有三个长期痛点：线程会阻塞，回调会割裂控制流，后台任务很容易脱离页面生命周期。

Android 的主线程负责绘制、事件分发和大部分 UI 状态更新。任何耗时操作，只要占住主线程太久，就会造成掉帧、卡顿甚至 ANR。于是我们把网络、数据库、文件读写放到后台线程。但线程 API 只解决了“在哪个线程执行”，没有解决“任务完成后如何回到原来的逻辑”“页面销毁后任务怎么办”“多个异步任务谁负责等待和取消”。协程真正补上的，就是这些控制权问题。

阻塞式写法：线程被占住 
fun loadUserBlocking(): User {
    Thread.sleep(1000)
    return api.loadUser()
}

这段代码的执行模型很直接：调用线程停在这里，什么也不能做。放在主线程会卡 UI；放在线程池里会占住一个工作线程。

挂起式写法：逻辑暂停，线程释放 
suspend fun loadUser(): User {
    delay(1000)
    return api.loadUser()
}

这段代码看起来仍然是顺序写法，但 delay 不会占住线程。协程保存“后面还要继续执行”的信息，当前线程可以去处理别的任务。

协程不是把异步变同步，而是让异步代码保持顺序表达

很多教程会说“协程让异步代码像同步代码一样写”。这句话有用，但容易误导。协程并没有把网络请求变成同步阻塞，它只是把“请求完成后继续执行什么”交给编译器和运行时管理。你看到的是顺序代码，底层仍然是挂起、回调、恢复、调度。

这就是为什么 suspend 函数既可以写得像普通函数，又不会阻塞线程。它把传统回调里的“下一步”变成了 Continuation。调用链不再被回调层层撕开，异常、取消、父子任务关系也可以被协程框架统一处理。

回调式：控制流被拆散 
api.loadUser(id, object : Callback<User> {
    override fun onSuccess(user: User) {
        api.loadOrders(user.id, object : Callback<List<Order>> {
            override fun onSuccess(orders: List<Order>) {
                render(user, orders)
            }
        })
    }
})

业务步骤被拆进多个回调，错误处理、取消、生命周期绑定都要额外设计。需求一复杂，很容易形成“回调树”。

协程式：业务步骤保持线性 
viewModelScope.launch {
    val user = repository.loadUser(id)
    val orders = repository.loadOrders(user.id)
    _state.value = UiState.Success(user, orders)
}

代码表达的是业务顺序：先用户，后订单，再更新状态。挂起和恢复由协程处理，生命周期由 viewModelScope 兜住。

把协程拆成三层理解

第一层是语法层。suspend 让函数可以挂起，launch 和 async 让你启动新的协程，Flow 让你描述一串异步数据。

第二层是运行时层。编译器生成状态机，Continuation 表示恢复点，Dispatcher 决定恢复在哪个线程，Job 维护任务状态、父子关系和取消传播。

第三层是架构层。在 Android 中，协程必须和 ViewModel、Lifecycle、Repository、UI 状态管理结合。否则你只是把回调换成了 launch，并没有真正解决生命周期和异常边界。

读后判断：如果你能解释“为什么 delay 不阻塞线程”“为什么 viewModelScope 能避免页面任务泄漏”“为什么协程代码看起来顺序但底层仍是回调恢复”，后面的状态机、Job 树和 Flow 才会真正串起来。

协程知识地图：把 API、源码和 Android 场景连起来

一句话抓住本质：协程把“回调后继续做什么”包装成 Continuation，并交给调度器、Job 树和上下文统一管理。

suspend

函数可能挂起。挂起时不占用线程；恢复时从上次保存的 label 继续执行。

CoroutineScope

协程的生命周期边界。Android 里常见的 viewModelScope、lifecycleScope 本质都是带 Job 的 Scope。

CoroutineContext

协程配置集合，常见元素包括 Job、Dispatcher、CoroutineName 和 CoroutineExceptionHandler。

Dispatcher

决定协程恢复时跑在哪个线程或线程池。它不是执行逻辑本身，而是调度策略。

Job

代表一个协程任务，维护状态、父子关系、取消、完成和异常传播。

Flow

异步数据流。冷流在 collect 时启动，热流独立存在并共享最新值或事件。

面试表达模板：协程是编译器、标准库和调度器配合实现的轻量级并发模型。编译器把 suspend 函数改写为状态机，标准库用 Continuation 表示恢复点，用 Job 维护结构化并发，用 Dispatcher 决定恢复线程。

suspend 与状态机：协程底层骨架

suspend 最容易被误解。它不代表“这个函数自动开线程”，也不代表“这个函数一定异步”。它只代表：这个函数可以在某个点暂停，并把后续执行封装成 Continuation。

把 suspend 理解成“函数可以被切成多段执行”会更准确。普通函数一旦开始执行，要么一路跑到 return，要么抛异常退出。挂起函数不同：它可以先执行前半段，遇到挂起点时把当前状态保存起来，然后立刻返回一个特殊标记。等异步操作完成后，再从保存的位置继续执行后半段。

这个过程最容易混淆的地方是“暂停”的对象。协程暂停的不是线程，而是这一段计算逻辑。线程只是执行代码的工人；当协程挂起时，工人不需要站在原地等，可以去执行别的任务。等结果来了，协程运行时再安排某个合适的线程来恢复这段计算。

一个挂起函数会被拆成哪些东西

suspend fun loadProfile(id: Long): Profile {
    val user = api.user(id)       // 挂起点 1
    val score = api.score(user)   // 挂起点 2
    return Profile(user, score)
}

编译器看到两个挂起点后，会把函数拆成三个执行片段：调用 api.user 之前、api.user 恢复之后到 api.score 之前、api.score 恢复之后到 return。为了知道下一次应该从哪里继续，它会生成类似 label 的状态字段。

同时，局部变量也不能简单放在普通栈帧里。因为挂起时当前函数已经返回了，原来的调用栈不再持续占用线程栈。像上面的 user，在第二个挂起点之后还要用，所以它会被提升到状态机对象的字段里。恢复时，状态机从字段中取回这些局部变量，继续执行后面的代码。

为什么返回值会变成 Any?

挂起函数编译后要表达两种完全不同的结果：如果没有挂起，它可以直接返回真实结果；如果挂起了，它要告诉调用方“我先返回，后续通过 Continuation 恢复”。所以编译后的函数通常会有一个宽泛的返回类型：正常完成时返回业务值，挂起时返回 COROUTINE_SUSPENDED。

关键点：COROUTINE_SUSPENDED 不是异常，也不是失败。它只是协程运行时内部使用的一个信号：当前函数没有同步完成，请不要继续当成普通返回值处理。

编译前后长什么样

// 你写的代码
suspend fun loadUser(id: Long): User {
    delay(300)
    return api.user(id)
}

// 编译后的心智模型，不是精确源码
fun loadUser(id: Long, cont: Continuation<User>): Any? {
    // 正常完成：返回 User
    // 发生挂起：返回 COROUTINE_SUSPENDED
}

Continuation 只有两个关键点：它携带 CoroutineContext，并通过 resumeWith(Result<T>) 恢复调用方。

public interface Continuation<in T> {
    public val context: CoroutineContext
    public fun resumeWith(result: Result<T>)
}

状态机的执行顺序

  1. 首次进入函数，label = 0。
  2. 遇到 delay、网络请求等挂起点。
  3. 保存下一次恢复位置：label = 1。
  4. 把当前 Continuation 注册给异步操作。
  5. 返回 COROUTINE_SUSPENDED，当前线程继续做别的事。
  6. 异步操作完成后调用 resumeWith。
  7. 状态机从 label = 1 继续执行。

常见误区

  • suspend 函数不会自动切线程，切线程靠 withContext 或调用方的 Dispatcher。
  • suspend 函数可以同步返回，也可以挂起后返回。
  • 挂起不是阻塞，阻塞会占住线程，挂起会释放线程。

  • COROUTINE_SUSPENDED 是异常吗？
  • 局部变量在挂起前后如何保存？
  • Continuation 和回调有什么关系？

Scope / Context / Dispatcher：协程的控制面

协程 API 看起来很多，但控制面可以压缩成三句话：Scope 管生命周期，Context 装配置，Dispatcher 管线程调度。

如果把协程看成一个任务，那么 Scope、Context、Dispatcher 分别回答三个不同问题：这个任务属于谁，带着哪些配置运行，恢复时应该交给哪个线程执行。很多协程 bug 本质上就是这三个问题没有回答清楚。

CoroutineScope 不只是一个“能 launch 的对象”。它代表任务的归属关系。一个协程从哪个 Scope 启动，就意味着它默认应该跟随这个 Scope 的生命周期。ViewModel 中的页面任务用 viewModelScope，是因为页面逻辑应该在 ViewModel 清理时结束；Repository 不应该偷用这个 Scope，是因为数据层不应该绑定某个页面。

CoroutineContext 则像一组不可变配置。它可以包含 Job、Dispatcher、CoroutineName、CoroutineExceptionHandler 等元素。多个 Context 用 + 合并时，同类型 Key 后者会覆盖前者。比如 viewModelScope.launch(Dispatchers.IO) 并不是创建了一个全新的生命周期，而是在原有 Scope 的 Job 之上替换了 Dispatcher。

Dispatcher 选择错，问题不会总是立刻暴露

把阻塞 IO 放在 Main 上，问题通常很明显：页面卡顿。更隐蔽的是把 CPU 密集任务长期丢到 IO，或者在 Default 里执行大量阻塞式数据库/文件读写。前者会让 IO 线程池被计算任务占住，后者会让计算线程等待外部资源，最终都可能造成吞吐下降和响应变慢。

选择 Dispatcher 时不要只看“这段代码耗时吗”，还要看它为什么耗时：等待磁盘、网络、数据库这类外部资源，通常属于 IO；排序、解析、压缩、加密、图片处理这类吃 CPU 的工作，通常属于 Default；更新 UI 必须回到 Main。

调试方法：遇到协程性能问题时，先在日志里打印线程名和 CoroutineName，再确认任务是阻塞等待、CPU 计算，还是 UI 渲染。Dispatcher 选错时，堆栈经常看不出错，但线程使用会很不合理。

Dispatchers.Main

主线程，更新 UI。Android 中由主线程 Looper 支撑。不要在 Main 上执行阻塞 IO 或大量 CPU 计算。

Dispatchers.IO

面向阻塞 IO 的共享线程池。适合文件、数据库、阻塞式网络 SDK。不要把 CPU 密集任务长期丢到 IO。

Dispatchers.Default

面向 CPU 密集任务，线程数通常与 CPU 核心相关。适合排序、解析、加密、图片处理等计算。

class UserRepository(
    private val api: UserApi,
    private val ioDispatcher: CoroutineDispatcher = Dispatchers.IO
) {
    suspend fun loadUser(id: Long): User = withContext(ioDispatcher) {
        api.fetchUser(id)
    }
}
架构建议：Repository 或 UseCase 内部自己切到合适 Dispatcher，让调用方不用关心底层是网络、数据库还是 CPU 计算。为了测试可控，Dispatcher 应该可注入。

结构化并发与取消：Job 树决定生死

结构化并发解决的是“谁负责等待、谁负责取消、异常往哪儿传”的问题。没有它，异步任务会变成到处泄漏的后台工作。

如果只看 launch，协程很像“启动一个后台任务”。但真正让协程适合大型 Android 项目的，是结构化并发。它要求新任务必须挂在一个明确的作用域下面，作用域结束时，子任务也必须完成或取消。这样你才能回答三个工程问题：页面退出后任务还在不在？一个子任务失败会不会影响其他任务？调用方什么时候能确定所有异步工作都结束了？

在没有结构化并发的异步模型里，任务经常像断线风筝一样飞出去。一个网络请求可能在页面销毁后才回调，一个后台任务可能静默失败，一个父流程可能已经结束但子任务还在跑。协程的 Job 树就是为了把这些关系显式化：每个协程都有自己的 Job，子协程挂到父 Job 下，父子之间有明确的完成、取消和异常传播规则。

普通 Job 的默认语义：一起成功，或者一起失败

在普通 coroutineScope 中，多个子任务通常表达一个整体工作。比如加载一个详情页必须同时拿到商品、价格和库存，其中任何一个失败都意味着这个整体结果不可用。这时一个子协程抛异常，父协程会进入取消流程，其他兄弟协程也会被取消。这样可以避免“整体已经失败，但某些子任务还在后台继续跑”的状态。

Supervisor 的语义：兄弟任务互不拖累

但不是所有页面都要求整体失败。首页模块、推荐流、广告、未读数往往可以部分成功。这个时候用普通 Job 会过于激进：广告失败不应该取消用户信息。SupervisorJob 和 supervisorScope 提供的就是“子失败不向上传播取消父”的边界。它不是吞异常，而是改变失败传播策略；每个子任务仍然需要自己处理结果或错误。

容易误解：Supervisor 不是“异常安全模式”。它只是隔离子任务失败，未处理异常仍然要被处理。实际项目里经常配合 runCatching、局部 try/catch 或统一错误映射。

coroutineScope

suspend fun loadScreen() = coroutineScope {
    val user = async { repo.user() }
    val feed = async { repo.feed() }
    Screen(user.await(), feed.await())
}

任何子任务失败，整个 scope 失败，兄弟任务被取消。适合“结果必须全部成功”的场景。

supervisorScope

suspend fun loadCards() = supervisorScope {
    val profile = async { runCatching { repo.profile() } }
    val ads = async { runCatching { repo.ads() } }
    profile.await() to ads.await()
}

子失败不自动取消父和兄弟。适合“多个模块互不拖累”的页面。

取消是协作式，不是抢占式

suspend fun crunch(items: List<Item>) = withContext(Dispatchers.Default) {
    for (item in items) {
        ensureActive()       // 主动检查取消
        process(item)
        if (item.index % 100 == 0) yield()
    }
}
高频坑：cancel() 只是设置取消状态。协程需要在挂起点、ensureActive()、yield() 或检查 isActive 时响应。纯 CPU while 循环如果不主动检查，可能无法及时取消。

异常处理体系：launch、async、CEH 不能混着背

协程异常面试最爱问，因为很多错误代码能跑，但生命周期和错误边界全是隐患。

协程异常处理的难点，不在于有没有 try/catch，而在于你要先判断异常属于哪个边界：是当前业务可以恢复的错误，还是某个子任务失败导致整个父任务失败，还是已经逃逸到根协程的未捕获异常。不同边界对应的处理方式完全不同。

CoroutineExceptionHandler 经常被误用成“协程里的全局 catch”。实际上它更像最后的未捕获异常处理器，只会处理到达根协程、且没有被正常消费的异常。业务逻辑里能恢复的错误，比如接口返回失败、缓存读取失败、单个模块加载失败，仍然应该在业务边界用 try/catch、runCatching 或结果类型处理。

先区分 launch 和 async 的异常语义

launch 表达“启动一个不返回结果的任务”。如果它是根协程，未捕获异常会交给根的异常处理流程；如果它是子协程，异常会先影响父 Job。async 表达“启动一个未来会返回结果的任务”，异常会被保存在 Deferred 中，并在 await() 时重新抛出。

但这里有个常见误区：不能简单说“async 异常不会影响父协程”。如果 async 是普通父子结构里的子协程，它失败时自己的 Job 仍然失败，默认仍会取消父协程；只是调用者在 await() 时还会再次拿到这个异常。只有根 async 或 supervisor 边界里的 async，才更接近“异常等 await 消费”。

整体必须成功：coroutineScope 
suspend fun loadPage() = coroutineScope {
    val user = async { api.user() }
    val orders = async { api.orders() }

    Page(user.await(), orders.await())
}

任意一个子任务失败，整个页面加载失败，兄弟任务取消。这适合强依赖聚合。

允许部分失败：supervisorScope 
suspend fun loadHome() = supervisorScope {
    val feed = async { runCatching { api.feed() } }
    val ads = async { runCatching { api.ads() } }

    Home(feed.await().getOrDefault(emptyList()), ads.await().getOrNull())
}

每个模块自己处理错误，失败不会拖垮兄弟模块。这适合首页、聚合页和弱依赖模块。

CancellationException 不是错误，是协程控制信号

取消在协程里是一种正常完成路径。页面退出、用户刷新、新搜索关键词到来、父任务失败，都可能触发取消。框架用 CancellationException 传递这个信号，所以它默认不会被当成业务错误上报。

这也是为什么捕获 Throwable 或宽泛的 Exception 时要格外小心。如果你吞掉了 CancellationException，上层会以为任务已经正常处理完，取消传播就被截断了。很多“页面退出后请求还在跑”“旧搜索结果覆盖新结果”的问题，都和取消信号被误吞有关。

场景异常表现正确处理方式
根协程 launch未捕获异常会交给 CoroutineExceptionHandler 或线程未捕获异常处理器在根 Scope 配置 CEH 兜底
子协程 launch异常先取消父 Job，再由父向兄弟传播取消局部 try/catch 或使用 supervisor 边界
根协程 async异常保存在 Deferred 中，await() 时抛出；CEH 不兜这个异常对 await() try/catch
普通父子结构中的 async异常仍会使子 Job 失败，并默认取消父；await() 还会重新抛出需要隔离时放进 supervisorScope 或使用 SupervisorJob
CancellationException表示正常取消，默认不当作错误上报不要吞掉它；catch Throwable 时要重新抛出
 
// 错误：吞掉 CancellationException，会破坏取消
try {
    repository.sync()
} catch (t: Throwable) {
    logger.log(t)
}

// 正确：取消要继续向外传
try {
    repository.sync()
} catch (e: CancellationException) {
    throw e
} catch (t: Throwable) {
    logger.log(t)
}
面试表达模板：CoroutineExceptionHandler 只处理“未被消费的根协程异常”。它不是 try/catch 的替代品，也不会处理 async 已经封装到 Deferred 里的异常。业务可恢复错误应该在业务边界 catch，CEH 更像最后兜底日志。

源码导读：不要从文件开始读，要从问题开始读

协程源码很容易越读越散。更有效的方式是带着问题进入源码：挂起如何恢复、调度器在哪里介入、Job 如何传播取消、Flow 如何保证上下文一致。

问题 1：suspend 如何从“暂停”回到原位置？

  • Continuation：恢复点抽象，核心方法是 resumeWith。
  • BaseContinuationImpl：编译器状态机的基类，负责调用 invokeSuspend。
  • label 字段：记录下一次恢复应该进入哪个分支。

读完应能：看懂“挂起不是线程暂停，而是函数返回；恢复不是从头执行，而是进入状态机下一段”。

问题 2：Dispatcher 在哪里切线程？

  • ContinuationInterceptor：Context 里拦截 Continuation 的入口。
  • DispatchedContinuation：把恢复动作包成可调度任务。
  • dispatch(context, block)：真正把任务提交到目标线程或线程池。

读完应能：理解 withContext 和 launch(Dispatchers.IO) 并不是“代码魔法”，而是恢复点被调度器包装。

问题 3：取消和异常为什么会沿 Job 树传播？

  • JobSupport：维护 Job 状态、完成回调、父子关系。
  • childCancelled：子协程失败后，父协程是否被取消的关键路径。
  • SupervisorJob：通过改变子失败处理策略，实现故障隔离。

读完应能：能解释普通 Job 和 SupervisorJob 的差异不是“try/catch 不同”，而是父子失败传播策略不同。

问题 4：Flow 如何保证上下文和顺序？

  • SafeCollector：检查 emit 是否发生在合法上下文。
  • AbstractFlow.collect：收集时包装 collector，并进入上游执行。
  • flowOn：在上下游之间建立协程边界，通常伴随 channel 传递。

读完应能：能说明为什么 catch 只捕获上游、flowOn 只影响上游、flow {} 里乱切上下文会出问题。

读源码顺序建议：先用一段最小代码制造现象，再沿调用链看 3 到 5 个关键类。不要试图从协程库入口通读全库；那样很快会迷路。

Flow：冷流、操作符链和 flowOn

Flow 是协程世界里的异步数据流。它的难点不是 API 名字，而是“谁触发生产、谁在哪个线程执行、异常从哪里被 catch”。

Flow 最适合用“管道”来理解，但这个管道有一个非常重要的特性：默认情况下，它不是提前运行的后台任务，而是一段等待被收集的声明。你写下 flow { emit(...) } 时，并没有立刻请求网络，也没有立刻发送数据。只有当某个协程调用 collect，上游 block 才开始执行。

这就是“冷流”的本质。冷流像一个函数，每次调用都会重新执行；热流像一个持续存在的状态源或事件源，是否有人订阅并不决定它是否存在。理解这一点，才能正确区分 Repository 返回 Flow、ViewModel 暴露 StateFlow、UI 收集状态这几层职责。

Flow 操作符不是列表转换，而是 collector 包装

很多人会把 Flow 的 map、filter 理解成集合操作符。语法上很像，但运行方式不同。集合是已经在内存里的数据；Flow 是一个异步发射过程。每个操作符会返回一个新的 Flow，它内部包装了下游 collector。等最终 collect 被调用时，这些包装才串成一条链。

所以数据方向和调用方向是相反的：collect 调用会从末端一路向上游建立链；emit 发生时，值再从上游一路经过 map、filter、catch、onEach 到达最终 collect。默认没有 buffer 时，上一个值要等下游处理完成，下一个值才会继续发射。

flowOn 为什么只影响上游

flowOn 不是“从这一行开始切线程”，而是“把我上游这段 Flow 放到指定上下文执行”。下游 collect 仍然运行在收集它的协程上下文里。这个设计让线程边界非常清晰：数据生产和转换可以在 IO 或 Default，UI 渲染仍然回到 Main。

当上下游上下文不同，Flow 通常需要在中间建立一个边界：上游在一个协程里发射，下游在另一个协程里接收，中间通过 channel 或类似机制传递值。这也是为什么 flowOn 会影响取消、异常和缓冲行为，不能把它当成普通函数调用里的 withContext。

读后判断：看到一条 Flow 链时，先标出三个边界：冷流从哪里开始执行，flowOn 影响到哪里，catch 能捕获哪一段。大多数 Flow bug 都能从这三件事里定位。

Cold Flow

每次 collect 都重新执行上游 block。适合网络轮询、数据库查询、一次性数据生产。

val users = flow {
    emit(api.fetchUsers())
}

Hot Flow

独立于收集者存在，多个收集者共享同一份数据。典型代表是 StateFlow 和 SharedFlow。

private val _state = MutableStateFlow(UiState())
val state = _state.asStateFlow()

操作符链是“从尾到头 collect，从头到尾 emit”

flow {
    emit(1)
    emit(2)
}.map { it * 10 }
 .filter { it > 10 }
 .collect { println(it) }

调用 collect 后，下游 collector 一层层包装上游；当原始 flow 执行 emit 时，数据沿着 map、filter、collect 传下去。默认情况下它是顺序管道，上一个值没处理完，不会处理下一个值。

flowOn 只影响上游

repository.userFlow()
    .map { heavyMap(it) }        // Dispatchers.Default
    .flowOn(Dispatchers.Default)
    .onEach { render(it) }       // Main，取决于 collect 所在协程
    .launchIn(viewModelScope)
不要在 flow {} 里随意 withContext 发射数据。Flow 有上下文保持规则，上游线程切换应该通过 flowOn 表达，这样异常、取消和背压语义更清晰。

StateFlow / SharedFlow：UI 状态和一次性事件分开建模

Android 页面里最常见的错误，是把状态和事件都塞进一个流里。状态要可重放、可读当前值；事件通常只想消费一次。

在 Android 页面中，状态和事件看起来都像“要通知 UI 的东西”，但它们的语义完全不同。状态描述的是“当前页面是什么样子”，例如是否加载中、列表数据是什么、错误提示文案是什么。事件描述的是“发生了一次动作”，例如跳转、弹 Toast、显示 Snackbar、请求权限。

StateFlow 适合状态，因为它总有一个当前值，新订阅者一上来就能拿到最新状态。旋转屏幕、页面重建、Compose 重组时，这个特性非常有用：UI 不需要等下一次网络回调，就能根据当前状态恢复界面。

但正因为 StateFlow 会保留最新值，它不适合表达一次性事件。如果你把 NavigateToDetail 放进 StateFlow，新 UI 订阅后可能再次收到这个旧值，导致重复跳转。很多项目用“消费后把事件置空”来补救，但这会把事件生命周期塞回状态对象里，复杂页面很快变得混乱。

状态：可以被重复观察 
data class LoginState(
    val loading: Boolean = false,
    val account: String = "",
    val error: String? = null
)

无论 UI 什么时候订阅，都应该能得到当前状态。旧状态被新 UI 重新渲染是合理的。

事件：只应该被消费一次 
sealed interface LoginEvent {
    data object NavigateHome : LoginEvent
    data class Toast(val message: String) : LoginEvent
}

导航和 Toast 是一次性动作。新 UI 不应该因为重新订阅就重复执行旧事件。

StateFlow 的 equals 去重既是优点也是陷阱

MutableStateFlow.value 更新时，如果新旧值按 equals 判断相等，收集者不会收到新值。这和不可变 data class 很搭：字段不变就不触发无意义刷新。但如果你在原地修改可变 List，再把同一个对象塞回 StateFlow，可能因为引用或 equals 语义导致 UI 不更新。

更稳的写法是让 UI 状态不可变，更新时用 copy 创建新对象，集合也尽量使用新列表。这样状态变化清晰，Compose 重组、DiffUtil、日志追踪都会更可控。

类型特点适合场景注意点
StateFlow必须有初始值，只保留最新值，按 equals 去重页面 UI 状态、表单状态、加载状态用不可变 data class，更新用 copy
SharedFlow可配置 replay、buffer、溢出策略导航、Toast、Snackbar、埋点事件一次性事件通常 replay = 0
Channel点对点通信，receive 消费后移除严格单消费者任务队列配置容量和关闭语义要谨慎
 
data class ProfileUiState(
    val loading: Boolean = false,
    val user: User? = null,
    val error: String? = null
)

class ProfileViewModel(
    private val repo: UserRepository
) : ViewModel() {
    private val _state = MutableStateFlow(ProfileUiState())
    val state: StateFlow<ProfileUiState> = _state.asStateFlow()

    private val _events = MutableSharedFlow<UiEvent>()
    val events: SharedFlow<UiEvent> = _events.asSharedFlow()

    fun load() = viewModelScope.launch {
        _state.update { it.copy(loading = true, error = null) }
        runCatching { repo.loadUser() }
            .onSuccess { user -> _state.update { it.copy(loading = false, user = user) } }
            .onFailure { e ->
                if (e is CancellationException) throw e
                _state.update { it.copy(loading = false, error = e.message) }
                _events.emit(UiEvent.ShowToast("加载失败"))
            }
    }
}
Compose 侧建议：使用生命周期感知的收集方式，例如 collectAsStateWithLifecycle()。传统 View 侧使用 repeatOnLifecycle，避免页面不可见时仍然收集 UI 数据。

Android 落地架构：让协程待在正确的层

稳定的协程代码不是“哪里都 launch 一下”，而是每一层只承担自己的职责。

Android 项目里，协程最容易被滥用的地方是 Scope 边界。很多问题不是 API 用错，而是协程启动在了错误的层：Repository 持有页面 Scope，Fragment 里直接开长期任务，ViewModel 暴露可变 StateFlow，UI 不按生命周期收集 Flow。代码短期能跑，长期就会出现泄漏、重复请求、状态错乱。

一个清晰的分层原则是：UI 层负责展示和用户输入，ViewModel 层负责把用户意图转换成状态变化，Data 层负责提供数据能力。Data 层不应该知道页面什么时候销毁，UI 层也不应该知道网络请求该切到哪个 Dispatcher。

Repository 暴露能力，不持有页面生命周期

Repository 更适合暴露 suspend 函数或 Flow。一次性请求用 suspend，持续数据源用 Flow。它可以在内部切到 IO，也可以组合缓存和网络，但不应该直接使用 viewModelScope。谁调用它，谁决定这个任务属于哪个生命周期。

ViewModel 启动页面任务，并把结果变成 UI 状态

ViewModel 是页面协程最常见的启动点。它用 viewModelScope 承接页面生命周期，用私有 MutableStateFlow 管状态，对外只暴露只读 StateFlow。异常在这里被转换成错误状态或一次性事件，而不是直接把 Throwable 扔给 UI。

UI 按生命周期收集，不保存业务协程

Fragment 或 Activity 收集 Flow 时，应该跟随可见生命周期启动和取消。传统 View 使用 repeatOnLifecycle，Compose 使用生命周期感知的收集 API。这样页面不可见时，UI 收集会停止，上游也有机会因为没有订阅者而停下来。

UI 层

  • 只收集状态、触发用户意图。
  • 不要在 Activity/Fragment 里长期持有裸 Scope。
  • 按生命周期收集 Flow。

ViewModel 层

  • 用 viewModelScope 启动页面任务。
  • 暴露不可变 StateFlow。
  • 把异常转成 UI 状态或事件。

Data 层

  • 优先暴露 suspend 或 Flow。
  • 内部切到 IO 或 Default。
  • 不要依赖 Android UI 生命周期。
// Fragment / View
viewLifecycleOwner.lifecycleScope.launch {
    repeatOnLifecycle(Lifecycle.State.STARTED) {
        viewModel.state.collect { state ->
            render(state)
        }
    }
}
为什么要 repeatOnLifecycle：它会在生命周期进入目标状态时启动收集，离开时取消收集。这样 UI 不可见时，上游能按取消语义停下来，减少泄漏和无意义刷新。

一套可复用的加载模式

fun refresh() = viewModelScope.launch {
    _state.update { it.copy(refreshing = true, error = null) }
    try {
        val data = repository.refresh()
        _state.update { it.copy(refreshing = false, items = data) }
    } catch (e: CancellationException) {
        throw e
    } catch (e: IOException) {
        _state.update { it.copy(refreshing = false, error = "网络异常") }
    } catch (e: Throwable) {
        _state.update { it.copy(refreshing = false, error = "未知错误") }
    }
}

场景实验室：把协程放回真实业务

真正能拉开水平差距的不是会写 launch，而是面对页面加载、搜索、部分失败、退出取消、事件消费时，能选对结构并解释原因。

下面的场景不是为了展示“某个 API 的标准写法”，而是训练判断能力。同样是并发请求，有的场景要整体失败，有的场景要部分可用；同样是 Flow，有的场景要保留所有值，有的场景只要最新值。API 选择取决于业务语义，而不是哪个写法看起来更高级。

读每个案例时建议先问三件事：第一，这些任务之间是强依赖还是弱依赖；第二，旧任务是否应该被新任务取消；第三，结果应该进入持续状态还是一次性事件。能回答这三件事，协程结构基本就不会选错。

Case 1

首页多个模块并发加载，部分失败不拖垮全页

业务诉求：用户信息、推荐流、广告、未读数可以并发加载。推荐流失败不应该让用户信息也消失。

suspend fun loadHome(): HomeUiModel = supervisorScope {
    val profile = async { runCatching { repo.profile() } }
    val feed = async { runCatching { repo.feed() } }
    val badges = async { runCatching { repo.badges() } }

    HomeUiModel(
        profile = profile.await().getOrNull(),
        feed = feed.await().getOrElse { emptyList() },
        badges = badges.await().getOrDefault(0)
    )
}

判断标准：多个模块互不强依赖，用 supervisorScope 做故障隔离；如果任何模块失败都必须整体失败，才用 coroutineScope。

Case 2

搜索框防抖，旧请求自动取消

业务诉求：用户持续输入时，不应该每个字符都请求；新关键词到来后，旧关键词的网络结果不应该覆盖新结果。

val results = queryFlow
    .debounce(300)
    .map { it.trim() }
    .distinctUntilChanged()
    .flatMapLatest { keyword ->
        if (keyword.isBlank()) flowOf(emptyList())
        else repository.search(keyword)
    }
    .stateIn(viewModelScope, SharingStarted.WhileSubscribed(5_000), emptyList())

判断标准：flatMapLatest 表达“只要最新”，旧上游会被取消；如果要保留所有请求顺序，才考虑 flatMapConcat。

Case 3

页面退出后，网络请求必须取消

业务诉求：详情页退出后，请求不应该继续更新已销毁页面；ViewModel 清理时任务要一起结束。

class DetailViewModel(
    private val repo: DetailRepository
) : ViewModel() {
    fun load(id: Long) = viewModelScope.launch {
        _state.update { it.copy(loading = true) }
        val detail = repo.detail(id)
        _state.update { it.copy(loading = false, detail = detail) }
    }
}

判断标准：页面任务用 viewModelScope，UI 收集用生命周期感知 API。不要把页面请求丢进 GlobalScope。

Case 4

一次性事件不要伪装成状态

业务诉求：Toast、导航、Snackbar 只消费一次，旋转屏幕或重新订阅时不应重复弹出。

private val _events = MutableSharedFlow<UiEvent>(
    replay = 0,
    extraBufferCapacity = 1
)
val events = _events.asSharedFlow()

fun submit() = viewModelScope.launch {
    runCatching { repo.submit() }
        .onSuccess { _events.emit(UiEvent.NavigateBack) }
        .onFailure { _events.emit(UiEvent.Toast("提交失败")) }
}

判断标准：状态用 StateFlow，事件用 SharedFlow。不要把导航事件塞进 UiState 后靠手动清空硬凑。

测试与调试：控制 Dispatcher 和虚拟时间

协程测试的关键是可控：不要让测试依赖真实线程、真实 delay、真实主线程。

协程测试最怕“偶尔过、偶尔不过”。如果测试依赖真实线程调度、真实 delay、真实主线程，就会变得慢且不稳定。好的协程代码从设计时就应该考虑可测试性：Dispatcher 可替换，时间可推进，状态可观察，异常路径可断言。

runTest 的价值在于提供一个测试用的协程环境和虚拟时间。代码里写 delay(5_000)，测试不需要真的等 5 秒；你可以用测试调度器推进时间，或者用 advanceUntilIdle() 让当前队列中的协程任务跑完。这样测试既快，又能稳定复现执行顺序。

为什么硬编码 Dispatchers.IO 会让测试难写

如果 Repository 内部直接写死 Dispatchers.IO，测试时这段代码会跑到真实 IO 线程池。你无法精确控制它什么时候执行，也很难让虚拟时间接管它。更好的方式是把 Dispatcher 当作依赖注入：生产环境传 Dispatchers.IO，测试环境传 StandardTestDispatcher。

难测试：调度器写死 
class UserRepo(private val api: Api) {
    suspend fun load() = withContext(Dispatchers.IO) {
        api.load()
    }
}

测试无法接管 IO 线程池，执行时机不稳定。

可测试：调度器注入 
class UserRepo(
    private val api: Api,
    private val io: CoroutineDispatcher
) {
    suspend fun load() = withContext(io) { api.load() }
}

测试可以传入 TestDispatcher，用虚拟时间和任务队列控制执行。

测试 ViewModel 时，应该断言状态流转

ViewModel 测试不应该只断言“调用了某个方法”。更有价值的是断言 UI 状态如何变化：初始状态是什么，触发加载后是否进入 loading，成功后是否有数据，失败后是否出现错误状态，取消时是否没有错误提示。这样测试覆盖的是用户真正能感知的行为。

测试 Flow 时，也不要只测第一个值。很多 bug 出现在第二次、第三次发射：防抖是否生效、旧请求是否被 flatMapLatest 取消、stateIn 是否重复订阅上游、错误是否被转换成空列表或错误状态。这些都需要按时间推进和多次收集来验证。

测试原则

  • Dispatcher 从构造函数注入。
  • 测试使用 runTest。
  • 用 StandardTestDispatcher 控制任务推进。
  • 用 advanceUntilIdle() 等待协程完成。

调试原则

  • 给关键协程加 CoroutineName。
  • 日志里打印线程名和 Job 状态。
  • 遇到卡死先看是否阻塞 Main。
  • 遇到泄漏先看 Scope 生命周期。
@Test
fun loadUser_updatesState() = runTest {
    val dispatcher = StandardTestDispatcher(testScheduler)
    val vm = ProfileViewModel(
        repo = FakeUserRepository(),
        dispatcher = dispatcher
    )

    vm.load()
    advanceUntilIdle()

    assertThat(vm.state.value.user?.name).isEqualTo("Ada")
}
面试高频：为什么不建议在代码里硬编码 Dispatchers.IO？因为测试无法替换，虚拟时间无法控制，也不利于在不同平台或场景下调整调度策略。

性能与反模式：协程不是万能加速器

协程让异步代码更好组织，但不会让阻塞代码自动变快。错误使用 Dispatcher、Flow 和 Scope，反而会制造卡顿、泄漏和难以复现的异常。

协程经常被误解成“轻量级线程”，于是有人以为多开协程就能提升性能。实际上，协程降低的是异步代码的组织成本和等待成本，不会让 CPU 计算变少，也不会让阻塞 IO 自动变快。如果底层 API 本身阻塞，协程只是帮你把阻塞放到更合适的线程池里；如果算法本身很重，协程也不能替你省掉计算。

排查协程性能问题，可以从四个方向入手：线程是否被阻塞，取消是否及时生效，Flow 是否产生过多中间值，生命周期是否导致重复收集或重复请求。很多页面卡顿并不是某个函数特别慢，而是这些小问题叠加起来：输入框每个字符都请求、旧请求没取消、列表每次发射都全量刷新、页面不可见时仍然收集。

反模式 1：用协程包装阻塞，却没有切 Dispatcher

suspend 本身不保证非阻塞。如果你在 suspend 函数里直接调用阻塞式 SDK，而调用方又在 Main 上执行，就仍然会卡主线程。正确做法是在数据层明确切到 IO，或者使用真正非阻塞的挂起 API。

反模式 2：Flow 发射太频繁，下游处理太重

Flow 默认是顺序的，但这不代表它天然高效。搜索输入、滚动位置、传感器数据、数据库变更都可能高频发射。如果每个值都触发网络请求、复杂 diff 或全页面状态重建，就会造成抖动。此时应该根据语义选择 debounce、sample、distinctUntilChanged、conflate 或 flatMapLatest。

反模式 3：共享上游没有共享，导致重复请求

冷流每次 collect 都会重新执行。如果两个 UI 区域分别 collect 同一个网络 Flow，可能触发两次请求。对于昂贵上游，应在 ViewModel 中用 stateIn 或 shareIn 共享结果，并用 SharingStarted.WhileSubscribed 控制没人订阅时是否停掉上游。

反模式清单

  • GlobalScope.launch 启动业务任务，生命周期无人管理。
  • 在 Main 线程执行阻塞 IO 或大量 JSON 解析。
  • 用 runBlocking 包住 Android UI 代码。
  • catch Throwable 后吞掉 CancellationException。
  • 把一次性事件建模成 StateFlow，旋转屏幕后重复消费。
  • 对高频 Flow 每个值都刷新整个页面。

优化抓手

  • CPU 任务放 Default，阻塞 IO 放 IO。
  • 高频输入使用 debounce、distinctUntilChanged、conflate。
  • 多个独立请求用 async 并发，但用结构化边界等待。
  • 共享昂贵上游用 stateIn 或 shareIn。
  • 长循环定期 yield 或 ensureActive。
searchText
    .debounce(300)
    .distinctUntilChanged()
    .flatMapLatest { keyword -> repository.search(keyword) }
    .catch { e -> emit(emptyList()) }
    .stateIn(
        scope = viewModelScope,
        started = SharingStarted.WhileSubscribed(5_000),
        initialValue = emptyList()
    )

练习题：看懂不算会，能判断结果才算稳

下面这些题适合读完对应章节后自测。不要急着展开答案，先说出现象，再说底层原因，最后给出修复方案。

flow {
    emit(1)
    emit(2)
}.map {
    println("map $it")
    it * 10
}.collect {
    println("collect $it")
}

输出顺序是 map 1、collect 10、map 2、collect 20。默认 Flow 是顺序管道，emit(1) 会等下游 map 和 collect 都处理完，才继续 emit(2)。

val job = viewModelScope.launch(Dispatchers.Default) {
    while (true) {
        calculateNext()
    }
}
job.cancel()

取消是协作式，不是抢占式。这个循环没有挂起点，也没有检查取消状态，所以可能持续运行。修复方式是在循环中使用 ensureActive()、yield() 或检查 isActive。

try {
    repository.sync()
} catch (t: Throwable) {
    report(t)
}

它会吞掉 CancellationException，破坏取消传播。正确写法是先捕获取消并重新抛出，再处理业务异常：catch (e: CancellationException) { throw e }。

flowOf(1)
    .catch { emit(0) }
    .collect {
        error("render failed")
    }

catch 只包住它上游的异常。collect 块属于下游，因此这里的异常不会被前面的 catch 捕获。要处理 collect 内部异常，应在 collect 外层使用 try/catch，或者把可能失败的逻辑移动到上游操作符中。

StateFlow 表达“当前状态”，会持有最新值并在新订阅者收集时立即发出。导航、Toast、Snackbar 属于一次性事件，重新订阅后重复消费会造成重复跳转或重复提示。更合适的是 SharedFlow(replay = 0) 或明确的事件通道。

// 不推荐
suspend fun load() = withContext(Dispatchers.IO) { api.load() }

// 推荐
class Repo(
    private val api: Api,
    private val io: CoroutineDispatcher
) {
    suspend fun load() = withContext(io) { api.load() }
}

生产代码注入 Dispatchers.IO，测试代码注入 StandardTestDispatcher 或 UnconfinedTestDispatcher。这样测试可以用虚拟时间和 advanceUntilIdle() 控制协程推进。

大厂面试题库：回答要有“现象 + 原理 + 实战边界”

下面这些题覆盖 Android 协程面试的大部分核心盘问。点击展开答案，建议自己先口述一遍，再对照补齐关键词。

编译器会把 suspend 函数改写成带 Continuation 参数的状态机。每个挂起点对应一个 label，挂起前保存局部变量和恢复位置，返回 COROUTINE_SUSPENDED；异步操作完成后通过 resumeWith 重新进入状态机。
阻塞会占住当前线程等待结果；挂起会保存协程状态并返回，当前线程可以执行其他任务。恢复时由 Continuation 和 Dispatcher 决定在哪里继续执行。
launch 返回 Job，表达“只关心完成和取消”；async 返回 Deferred，表达“未来会有结果”。launch 根协程未捕获异常会走 CEH；async 的异常会保存在 Deferred 中，await 时抛出。但在普通父子结构中，async 失败仍会默认取消父 Job。
CEH 只处理根协程未捕获异常。它不会处理子协程局部异常、不会替代 try/catch、不会处理 async 已保存到 Deferred 的异常，也不会处理 CancellationException。
普通 Job 中子协程失败会取消父协程，父再取消兄弟协程。SupervisorJob 隔离子失败，子异常不自动取消父和兄弟，适合页面多个模块互不影响的加载场景。
withContext 是挂起函数，会切换上下文并等待代码块完成，返回结果；launch 会启动一个新协程，立即返回 Job。需要顺序拿结果用 withContext，需要并发任务用 launch/async。
flow{} 的 block 只有 collect 时才执行；每次 collect 都重新执行一遍，收集者之间默认互不共享。所以它适合描述数据生产逻辑，而不是保存 UI 状态。
flowOn 只影响它上游的 Flow 及操作符，不影响下游 collect。它通常通过上下游协程和 channel 边界实现上下文切换，同时保持 Flow 的上下文规则。
catch 捕获的是上游异常，包括 emit 和上游操作符抛出的异常。collect 块里的异常属于下游，不会被前面的 catch 捕获，需要在 collect 外层 try/catch。
页面状态用 StateFlow，因为它有初始值、保留最新值、可同步读取 value、按 equals 去重。一次性事件用 SharedFlow，因为它可以不重放，避免旋转屏幕或重新订阅时重复消费。
它绑定 ViewModel 生命周期，ViewModel 清理时自动取消子协程；底层通常是 SupervisorJob + Main Dispatcher。这样页面任务不会泄漏到 ViewModel 之外。
它让 Flow 收集随着生命周期自动启动和取消。页面不可见时取消收集，重新可见时再次收集，避免 UI 不可见时仍刷新或泄漏。
GlobalScope 不绑定业务生命周期，任务失败和取消都难以管理，容易造成泄漏和后台任务失控。除非是进程级、明确生命周期的全局任务，否则应使用业务 Scope。
在 coroutineScope 中用 async 启动两个子任务，再 await 合并。这样外层会等待所有子任务；任何一个失败会取消另一个并把异常抛给调用方。如果希望部分失败可用，用 supervisorScope + runCatching。
CancellationException 是协程取消的正常信号。如果 catch Throwable 后吞掉它，父协程以为任务正常完成，取消传播被破坏，可能导致资源继续运行或状态错误。
使用 kotlinx-coroutines-test 的 runTest、TestDispatcher 和 TestScheduler。生产代码通过构造函数注入 Dispatcher，测试中替换为 StandardTestDispatcher，再用 advanceUntilIdle 或 advanceTimeBy 推进虚拟时间。

追问训练：把一句答案升级成一条推理链

大厂面试通常不会停在“是什么”。真正的追问会从 API 问到异常、取消、源码模型和项目取舍。下面用几条典型链路训练回答层次。

链路 A：suspend 到状态机

  1. 第一问：suspend 是什么？
  2. 追问：它会自动切线程吗？
  3. 追问：挂起时线程去哪了？
  4. 追问：局部变量如何恢复？
  5. 加分：说出 Continuation、label、COROUTINE_SUSPENDED 的关系。

链路 B：async 到异常边界

  1. 第一问：launch 和 async 区别？
  2. 追问：async 异常什么时候抛？
  3. 追问：CEH 为什么捕不到？
  4. 追问：子 async 失败会不会取消父？
  5. 加分：区分根 async 与结构化父子 async。

链路 C：Flow 到线程切换

  1. 第一问：Flow 为什么是冷流？
  2. 追问：操作符链执行顺序？
  3. 追问：flowOn 影响上游还是下游？
  4. 追问：为什么不推荐在 flow {} 里 withContext emit？
  5. 加分：说出上下文保持和 channel 边界的心智模型。

链路 D：Android 生命周期

  1. 第一问：viewModelScope 解决什么？
  2. 追问：为什么 UI 收集 Flow 要感知生命周期？
  3. 追问：repeatOnLifecycle 离开 STARTED 会发生什么？
  4. 追问：Repository 是否应该依赖 ViewModelScope？
  5. 加分：讲清 UI、ViewModel、Data 三层协程边界。

回答评分标准

60 分能说出 API 作用和基本用法，但容易混淆异常、取消和生命周期边界。 75 分能结合 Android 场景给出正确写法，知道 viewModelScope、repeatOnLifecycle、StateFlow 的适用场景。 90 分能用状态机、Job 树、Dispatcher、Flow 管道解释现象，并能指出常见错误用法。 95+ 分能主动补充源码心智模型、测试策略、性能边界和业务取舍，不靠背诵也能处理变体追问。
