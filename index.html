01 📖 入门准备 路线 · 类比 · 知识地图

学习路线：先会用，再讲透，最后能排坑

很多人学协程会卡在两个极端：只背 API，遇到异常和取消就崩；一上来啃源码，缺少 Android 使用场景。建议按下面顺序推进。

第一层：把协程当成“可取消的异步任务”

掌握 launch、async/await、withContext、viewModelScope，能把网络、数据库、文件 IO 从主线程挪出去。

第二层：理解 suspend 的真实形态

suspend 不创建线程，不等于异步。它只是允许函数在挂起点保存现场、返回、稍后恢复。

第三层：用 Job 树解释生命周期

父取消子、子异常影响父、兄弟协程如何被级联取消，这些都由结构化并发和 Job 状态机控制。

第四层：把 Flow 当成数据管道

冷流负责声明数据生产逻辑，热流负责状态共享。理解 flowOn、catch、stateIn 才能写稳定 UI。

第五层：面向 Android 架构落地

ViewModel 暴露状态，Repository 暴露 suspend/Flow，UI 按生命周期收集，测试里控制 Dispatcher 和虚拟时间。

为什么需要协程

学习协程之前，先不要急着背 launch、async、withContext。更重要的问题是：Android 开发为什么会需要一种新的异步模型？答案不是“线程太重”这么简单，而是传统异步代码有三个长期痛点：线程会阻塞，回调会割裂控制流，后台任务很容易脱离页面生命周期。

Android 的主线程负责绘制、事件分发和大部分 UI 状态更新。任何耗时操作，只要占住主线程太久，就会造成掉帧、卡顿甚至 ANR。于是我们把网络、数据库、文件读写放到后台线程。但线程 API 只解决了“在哪个线程执行”，没有解决“任务完成后如何回到原来的逻辑”“页面销毁后任务怎么办”“多个异步任务谁负责等待和取消”。协程真正补上的，就是这些控制权问题。

阻塞式写法：线程被占住 
fun loadUserBlocking(): User {
    Thread.sleep(1000)
    return api.loadUser()
}

这段代码的执行模型很直接：调用线程停在这里，什么也不能做。放在主线程会卡 UI；放在线程池里会占住一个工作线程。

挂起式写法：逻辑暂停，线程释放 
suspend fun loadUser(): User {
    delay(1000)
    return api.loadUser()
}

这段代码看起来仍然是顺序写法，但 delay 不会占住线程。协程保存“后面还要继续执行”的信息，当前线程可以去处理别的任务。

协程不是把异步变同步，而是让异步代码保持顺序表达

很多教程会说“协程让异步代码像同步代码一样写”。这句话有用，但容易误导。协程并没有把网络请求变成同步阻塞，它只是把“请求完成后继续执行什么”交给编译器和运行时管理。你看到的是顺序代码，底层仍然是挂起、回调、恢复、调度。

这就是为什么 suspend 函数既可以写得像普通函数，又不会阻塞线程。它把传统回调里的“下一步”变成了 Continuation。调用链不再被回调层层撕开，异常、取消、父子任务关系也可以被协程框架统一处理。

回调式：控制流被拆散 
api.loadUser(id, object : Callback<User> {
    override fun onSuccess(user: User) {
        api.loadOrders(user.id, object : Callback<List<Order>> {
            override fun onSuccess(orders: List<Order>) {
                render(user, orders)
            }
        })
    }
})

业务步骤被拆进多个回调，错误处理、取消、生命周期绑定都要额外设计。需求一复杂，很容易形成“回调树”。

协程式：业务步骤保持线性 
viewModelScope.launch {
    val user = repository.loadUser(id)
    val orders = repository.loadOrders(user.id)
    _state.value = UiState.Success(user, orders)
}

代码表达的是业务顺序：先用户，后订单，再更新状态。挂起和恢复由协程处理，生命周期由 viewModelScope 兜住。

把协程拆成三层理解

第一层是语法层。suspend 让函数可以挂起，launch 和 async 让你启动新的协程，Flow 让你描述一串异步数据。

第二层是运行时层。编译器生成状态机，Continuation 表示恢复点，Dispatcher 决定恢复在哪个线程，Job 维护任务状态、父子关系和取消传播。

第三层是架构层。在 Android 中，协程必须和 ViewModel、Lifecycle、Repository、UI 状态管理结合。否则你只是把回调换成了 launch，并没有真正解决生命周期和异常边界。

读后判断：如果你能解释"为什么 delay 不阻塞线程""为什么 viewModelScope 能避免页面任务泄漏""为什么协程代码看起来顺序但底层仍是回调恢复"，后面的状态机、Job 树和 Flow 才会真正串起来。

协程 vs 线程 vs RxJava：一张表看清谁解决什么问题

协程知识地图：把 API、源码和 Android 场景连起来

一句话抓住本质：协程把“回调后继续做什么”包装成 Continuation，并交给调度器、Job 树和上下文统一管理。

suspend

函数可能挂起。挂起时不占用线程；恢复时从上次保存的 label 继续执行。

CoroutineScope

协程的生命周期边界。Android 里常见的 viewModelScope、lifecycleScope 本质都是带 Job 的 Scope。

CoroutineContext

协程配置集合，常见元素包括 Job、Dispatcher、CoroutineName 和 CoroutineExceptionHandler。

Dispatcher

决定协程恢复时跑在哪个线程或线程池。它不是执行逻辑本身，而是调度策略。

Job

代表一个协程任务，维护状态、父子关系、取消、完成和异常传播。

Flow

异步数据流。冷流在 collect 时启动，热流独立存在并共享最新值或事件。

面试表达模板：协程是编译器、标准库和调度器配合实现的轻量级并发模型。编译器把 suspend 函数改写为状态机，标准库用 Continuation 表示恢复点，用 Job 维护结构化并发，用 Dispatcher 决定恢复线程。

类比破壁：把协程黑话翻译成生活场景

协程术语多，换个角度全变简单了。下面每个核心概念配一个生活类比，看完再回去读技术部分，会发现全通了。

Continuation = 点外卖的取餐号

你在 App 里下了单，拿到一个取餐号。你可以先去开会、写代码——不用在店门口站着等。饭好了，叫号器一响，你拿着号码回来取。Continuation 就是这个取餐号：它记住了你从哪暂停的（label），等你回来时接着往下执行（resumeWith）。

类比：如果不用取餐号，你只能站在店门口一直等——这就是"阻塞线程"。取餐号让你释放线程去做别的事——这就是"挂起不阻塞"。

CoroutineScope = 会议室的租约

你租了一间会议室（Scope），在里面开会（启动协程）。租约到期（Scope cancel），会议室里的所有人（子协程）必须散会——不管他们还有没有说完。这就是结构化并发：房间的租约就是所有在里面工作的任务的硬上限。

GlobalScope 就是"不签租约随便用"——会议室永远不关门，别人想进来就进来，极容易出事。

CoroutineContext = 行李箱上的行李牌

你的行李箱上贴满了各种标签：姓名（CoroutineName）、航班号（Job）、目的地（Dispatcher）、保险单（CEH）。每个标签独立贴在箱子上，统一管理。Context 就是所有标签的集合。+"就是把一张新标签贴上去，如果原来已经有同类的旧标签，新标签覆盖旧的。

Dispatcher = 快递公司的分拣中心

你有一个包裹（协程），Dispatcher 就是快递公司的分拣中心。它不自己送快递，只是决定包裹被哪个站点（线程）处理。Default 就是"CPU 处理中心"（按核心数分配人手），IO 就是"仓储物流中心"（人手弹性扩展），Main 就是"线下门店"（只有一个人，不能干重活）。

Job 树 = 公司组织架构

CEO（根 Job）下面有部门总监（子 Job），总监下有经理（孙 Job）。公司倒闭（CEO cancel），所有部门全部解散——取消向下传播。某个部门出了严重事故（子异常），CEO 有可能被连累下课——异常向上传播。

SupervisorJob 就是"给了部门独立运营权"——某个部门出事，不影响其他部门，也不影响 CEO。

CancellationException = 公司发的下班通知

公司 6 点下班是正常安排——不是开除你。CancellationException 就是这个下班通知：它不是错误，不会上报 HR（CEH），只是告诉你"今天到这，收拾东西走"（finally 块正常执行清理）。

RuntimeException 才是"你被开除了"——HR 部（CEH）要记录，所有跟你合作的项目（兄弟协程）也要停下来。

Flow 冷流 = Netflix 点播；热流 = 收音机

冷流（点播）：你不点播放，电影不会自己开始放。每个人在自己设备上点播，看到的是自己从头开始的独立播放。这就是 flow {}：不 collect 不执行，每个收集者独立。

热流（收音机）：电台一直在播，不管你有没有在听。你打开收音机听到的是当前正在播的内容，而不是从早上 6 点开始的录音。这就是 StateFlow/SharedFlow：数据源独立于监听者。

Channel = 工厂里的传送带

上游工人往传送带上放零件（send），下游工人从传送带末端取零件（receive）。传送带满了上游就停手等（挂起），传送带空了下游就等。RENDEZVOUS 就是"没有传送带——上游必须亲手递，下游必须亲手接"。UNLIMITED 就是"传送带无限长，上游随便放"。

Mutex = 只有一个坑的卫生间

卫生间门上有个"使用中"的牌子。你进去后翻牌子（lock），出来后翻回（unlock）。有人在里面你就在门口等——但你不是傻站着堵住走廊（阻塞），而是去旁边的沙发上坐着等（挂起），里面的人出来叫你一声你再站起来进去。

CEH = 前台的投诉记录本

公司各部门出了问题（子异常），先沿着汇报线层层上报。如果一路都没人处理，最后会到前台的意见簿上（CEH）。但前台不会替部门解决问题——它只是做好记录。而且，如果是正常下班（CancellationException），前台不会记。

怎么用这个专区：遇到某个概念读不懂时，先回来看对应的类比。脑中有生活场景，再去看技术定义，会发现"哦原来就是这个意思"。
02 ⚙️ 核心机制 suspend · Context · 结构化并发 · 异常

suspend 与状态机：协程底层骨架

suspend 最容易被误解。它不代表“这个函数自动开线程”，也不代表“这个函数一定异步”。它只代表：这个函数可以在某个点暂停，并把后续执行封装成 Continuation。

把 suspend 理解成“函数可以被切成多段执行”会更准确。普通函数一旦开始执行，要么一路跑到 return，要么抛异常退出。挂起函数不同：它可以先执行前半段，遇到挂起点时把当前状态保存起来，然后立刻返回一个特殊标记。等异步操作完成后，再从保存的位置继续执行后半段。

这个过程最容易混淆的地方是“暂停”的对象。协程暂停的不是线程，而是这一段计算逻辑。线程只是执行代码的工人；当协程挂起时，工人不需要站在原地等，可以去执行别的任务。等结果来了，协程运行时再安排某个合适的线程来恢复这段计算。

一个挂起函数会被拆成哪些东西

suspend fun loadProfile(id: Long): Profile {
    val user = api.user(id)       // 挂起点 1
    val score = api.score(user)   // 挂起点 2
    return Profile(user, score)
}

编译器看到两个挂起点后，会把函数拆成三个执行片段：调用 api.user 之前、api.user 恢复之后到 api.score 之前、api.score 恢复之后到 return。为了知道下一次应该从哪里继续，它会生成类似 label 的状态字段。

同时，局部变量也不能简单放在普通栈帧里。因为挂起时当前函数已经返回了，原来的调用栈不再持续占用线程栈。像上面的 user，在第二个挂起点之后还要用，所以它会被提升到状态机对象的字段里。恢复时，状态机从字段中取回这些局部变量，继续执行后面的代码。

为什么返回值会变成 Any?

挂起函数编译后要表达两种完全不同的结果：如果没有挂起，它可以直接返回真实结果；如果挂起了，它要告诉调用方“我先返回，后续通过 Continuation 恢复”。所以编译后的函数通常会有一个宽泛的返回类型：正常完成时返回业务值，挂起时返回 COROUTINE_SUSPENDED。

关键点：COROUTINE_SUSPENDED 不是异常，也不是失败。它只是协程运行时内部使用的一个信号：当前函数没有同步完成，请不要继续当成普通返回值处理。

编译前后长什么样

// 你写的代码
suspend fun loadUser(id: Long): User {
    delay(300)
    return api.user(id)
}

// 编译后的心智模型，不是精确源码
fun loadUser(id: Long, cont: Continuation<User>): Any? {
    // 正常完成：返回 User
    // 发生挂起：返回 COROUTINE_SUSPENDED
}

Continuation 只有两个关键点：它携带 CoroutineContext，并通过 resumeWith(Result<T>) 恢复调用方。

public interface Continuation<in T> {
    public val context: CoroutineContext
    public fun resumeWith(result: Result<T>)
}

状态机的执行顺序

  1. 首次进入函数，label = 0。
  2. 遇到 delay、网络请求等挂起点。
  3. 保存下一次恢复位置：label = 1。
  4. 把当前 Continuation 注册给异步操作。
  5. 返回 COROUTINE_SUSPENDED，当前线程继续做别的事。
  6. 异步操作完成后调用 resumeWith。
  7. 状态机从 label = 1 继续执行。

常见误区

  • suspend 函数不会自动切线程，切线程靠 withContext 或调用方的 Dispatcher。
  • suspend 函数可以同步返回，也可以挂起后返回。
  • 挂起不是阻塞，阻塞会占住线程，挂起会释放线程。

源码追踪：suspend 函数如何变成状态机

下面从编译器的角度，用一个完整的伪代码示例追踪 suspend fun → 状态机 → COROUTINE_SUSPENDED → resumeWith 的全链路。

一个极简挂起函数被编译后长什么样

// === 你写的源码 ===
suspend fun simpleSuspend(): String {
    delay(1000)
    return "done"
}

// === 编译器等价生成（简化伪代码） ===
class SimpleSuspendSM(
    private val completion: Continuation<String>
) : Continuation<Any?> {
    var label = 0
    var result: Any? = null
    override val context = completion.context

    override fun resumeWith(result: Result<Any?>) {
        this.result = result.getOrNull()
        val outcome = invokeSuspend()
        if (outcome != COROUTINE_SUSPENDED) {
            completion.resumeWith(Result.success(outcome as String))
        }
    }

    fun invokeSuspend(): Any? {
        when (label) {
            0 -> {
                label = 1
                val r = delay(1000, this)           // 把自己作为 Continuation 传入
                if (r == COROUTINE_SUSPENDED) return COROUTINE_SUSPENDED
            }
            1 -> { return "done" }
        }
        error("unreachable")
    }
}

执行流程六步追踪

① 首次调用

② 进入标签0

③ delay 返回 COROUTINE_SUSPENDED

④ 1 秒后定时器触发回调

⑤ resumeWith 驱动状态机

resumeWith 保存结果，再次调用 invokeSuspend() → label = 1 → 走标签1 → 返回 "done"。

⑥ 完成

核心理解：状态机对象 同时是 Continuation 和回调——delay 完成时回调的是它自己，它在 resumeWith 里再驱动状态机继续。这就是协程"挂起-恢复"循环的本质。

Scope / Context / Dispatcher：协程的控制面

协程 API 看起来很多，但控制面可以压缩成三句话：Scope 管生命周期，Context 装配置，Dispatcher 管线程调度。

如果把协程看成一个任务，那么 Scope、Context、Dispatcher 分别回答三个不同问题：这个任务属于谁，带着哪些配置运行，恢复时应该交给哪个线程执行。很多协程 bug 本质上就是这三个问题没有回答清楚。

CoroutineScope 不只是一个“能 launch 的对象”。它代表任务的归属关系。一个协程从哪个 Scope 启动，就意味着它默认应该跟随这个 Scope 的生命周期。ViewModel 中的页面任务用 viewModelScope，是因为页面逻辑应该在 ViewModel 清理时结束；Repository 不应该偷用这个 Scope，是因为数据层不应该绑定某个页面。

CoroutineContext 则像一组不可变配置。它可以包含 Job、Dispatcher、CoroutineName、CoroutineExceptionHandler 等元素。多个 Context 用 + 合并时，同类型 Key 后者会覆盖前者。比如 viewModelScope.launch(Dispatchers.IO) 并不是创建了一个全新的生命周期，而是在原有 Scope 的 Job 之上替换了 Dispatcher。

Dispatcher 选择错，问题不会总是立刻暴露

把阻塞 IO 放在 Main 上，问题通常很明显：页面卡顿。更隐蔽的是把 CPU 密集任务长期丢到 IO，或者在 Default 里执行大量阻塞式数据库/文件读写。前者会让 IO 线程池被计算任务占住，后者会让计算线程等待外部资源，最终都可能造成吞吐下降和响应变慢。

选择 Dispatcher 时不要只看“这段代码耗时吗”，还要看它为什么耗时：等待磁盘、网络、数据库这类外部资源，通常属于 IO；排序、解析、压缩、加密、图片处理这类吃 CPU 的工作，通常属于 Default；更新 UI 必须回到 Main。

调试方法：遇到协程性能问题时，先在日志里打印线程名和 CoroutineName，再确认任务是阻塞等待、CPU 计算，还是 UI 渲染。Dispatcher 选错时，堆栈经常看不出错，但线程使用会很不合理。

CoroutineContext 链表源码分析

面试如果只回答"Context 是配置集合"不够。源码告诉你它是一个 不可变的、用 Key 索引的链表，CombinedContext 是真实数据结构：

// 内部实现: CombinedContext (极简版)
internal class CombinedContext(
    private val left: CoroutineContext,     // 指向前一个节点
    private val element: Element             // 当前存的 Element
) : CoroutineContext {
    
    override fun <E : Element> get(key: Key<E>): E? {
        var cur = this
        while (cur is CombinedContext) {
            cur.element[key]?.let { return it }   // 当前节点匹配就返回
            cur = cur.left                         // 否则向左遍历
        }
        return cur[key]  // 到达单Element 或 Empty
    }
}
面试追问——为什么不用 HashMap?
① Element 极少（3-5个），链表 O(n) 开销可忽略
② 内存占用极小：每个节点只存 left+element 两个引用
③ 不可变链表插入 O(1)，+ 操作在头部追加即可

CoroutineDispatcher 与 DispatchedContinuation

Dispatcher 不是"主动调度协程"，而是被 ContinuationInterceptor 接口被动钩入协程恢复流程：

// 每次启动或恢复时, .intercepted() 找到 Dispatcher 把原始 Continuation 包一层
override fun <T> interceptContinuation(continuation: Continuation<T>): Continuation<T> =
    DispatchedContinuation(this, continuation)

// DispatchedContinuation 同时是 Continuation 和 Runnable 
class DispatchedContinuation<T>(
    val dispatcher: CoroutineDispatcher,
    val continuation: Continuation<T>
) : DispatchedTask<T>(...), Continuation<T> by continuation {
    
    override fun resumeWith(result: Result<T>) {
        if (dispatcher.isDispatchNeeded(context)) {
            dispatcher.dispatch(context, this)  // this=Runnable → 提交到线程池
        } else {
            executeUnconfined(...)              // Unconfined 直接执行
        }
    }
    
    override fun run() {
        continuation.resumeWith(takeState().toResult())  // 线程池执行的入口
    }
}
核心模型：DispatchedContinuation = 快递包裹。它既是 Continuation（对上层透明），又是 Runnable（可被线程池投递）。dispatch(context, this) 把自己扔进线程池 → 线程池调用 run() → 内部调用真正的 continuation.resumeWith。

结构化并发与取消：Job 树决定生死

结构化并发解决的是“谁负责等待、谁负责取消、异常往哪儿传”的问题。没有它，异步任务会变成到处泄漏的后台工作。

如果只看 launch，协程很像“启动一个后台任务”。但真正让协程适合大型 Android 项目的，是结构化并发。它要求新任务必须挂在一个明确的作用域下面，作用域结束时，子任务也必须完成或取消。这样你才能回答三个工程问题：页面退出后任务还在不在？一个子任务失败会不会影响其他任务？调用方什么时候能确定所有异步工作都结束了？

在没有结构化并发的异步模型里，任务经常像断线风筝一样飞出去。一个网络请求可能在页面销毁后才回调，一个后台任务可能静默失败，一个父流程可能已经结束但子任务还在跑。协程的 Job 树就是为了把这些关系显式化：每个协程都有自己的 Job，子协程挂到父 Job 下，父子之间有明确的完成、取消和异常传播规则。

普通 Job 的默认语义：一起成功，或者一起失败

在普通 coroutineScope 中，多个子任务通常表达一个整体工作。比如加载一个详情页必须同时拿到商品、价格和库存，其中任何一个失败都意味着这个整体结果不可用。这时一个子协程抛异常，父协程会进入取消流程，其他兄弟协程也会被取消。这样可以避免“整体已经失败，但某些子任务还在后台继续跑”的状态。

Supervisor 的语义：兄弟任务互不拖累

但不是所有页面都要求整体失败。首页模块、推荐流、广告、未读数往往可以部分成功。这个时候用普通 Job 会过于激进：广告失败不应该取消用户信息。SupervisorJob 和 supervisorScope 提供的就是“子失败不向上传播取消父”的边界。它不是吞异常，而是改变失败传播策略；每个子任务仍然需要自己处理结果或错误。

容易误解：Supervisor 不是“异常安全模式”。它只是隔离子任务失败，未处理异常仍然要被处理。实际项目里经常配合 runCatching、局部 try/catch 或统一错误映射。

coroutineScope

suspend fun loadScreen() = coroutineScope {
    val user = async { repo.user() }
    val feed = async { repo.feed() }
    Screen(user.await(), feed.await())
}

任何子任务失败，整个 scope 失败，兄弟任务被取消。适合“结果必须全部成功”的场景。

supervisorScope

suspend fun loadCards() = supervisorScope {
    val profile = async { runCatching { repo.profile() } }
    val ads = async { runCatching { repo.ads() } }
    profile.await() to ads.await()
}

子失败不自动取消父和兄弟。适合“多个模块互不拖累”的页面。

取消是协作式，不是抢占式

suspend fun crunch(items: List<Item>) = withContext(Dispatchers.Default) {
    for (item in items) {
        ensureActive()       // 主动检查取消
        process(item)
        if (item.index % 100 == 0) yield()
    }
}
高频坑：cancel() 只是设置取消状态。协程需要在挂起点、ensureActive()、yield() 或检查 isActive 时响应。纯 CPU while 循环如果不主动检查，可能无法及时取消。

源码追踪：JobSupport 状态机与取消传播链路

JobSupport 用一个 AtomicReference 字段同时承担状态标记、子节点链表头、锁三种角色。下面逐行拆解。

JobSupport._state——单字段三用

很多人以为 Job 的状态是一个简单的 enum。源码里它是一个 AtomicReference<Any?>——既存状态值，也存子节点链表：

// JobSupport (简化)
public open class JobSupport : Job, ChildJob, ParentJob {
    private val _state = AtomicReference<Any?>(EMPTY_ACTIVE)
    // _state 的可能值:
    // - EMPTY_ACTIVE / EMPTY_NEW    → 活跃，还没子节点
    // - IncompleteList / NodeList    → 活跃，有子节点(链表)
    // - CompletedExceptionally/...  → 完成状态
    // - Cancelling / Cancelled      → 取消状态
}

取消传播的递归入口：cancelImpl

// cancelImpl 核心逻辑
internal fun cancelImpl(cause: Any?): Boolean {
    var finalState: Any? = cause
    // 1. CAS 把状态改成 Cancelling; 如果已不可取消, 返回 false
    // 2. 遍历 children 链表, 对每个 childJob 递归调用 cancelImpl
    var child = firstChild(state)
    while (child != null) {
        child.childJob.cancelImpl(cause)   // 深度优先，一直到底
        child = child.nextSibling
    }
    return true
}

取消是深度优先的递归：父取消 → 遍历所有直接子 → 每个子取消它的子 → ... → 最底层的协程在下一个挂起点抛 CancellationException。

ChildHandleNode——父子连接的桥梁

internal class ChildHandleNode(
    private val parent: JobSupport,
    private val child: ChildJob
) : ChildHandle, CompletionHandler {
    // 父完成时的回调 → 如果父是被取消的, 通知子
    override fun invoke(cause: Throwable?) {
        child.parentCancelled(cause)
    }
    // 子完成/异常时的回调 → 如果子是非取消异常, 通知父
    override fun childCancelled(cause: Throwable?): Boolean {
        if (cause is CancellationException) return true  // 取消不传播
        return parent.childCancelled(cause)              // 真正异常→向上传播
    }
}
SupervisorJob 一言：覆写了 childCancelled 直接返回 false（"子异常不传播给父"），这就是 supervisorScope 能隔离故障的源码本源。

面试里经常被要求"画出 Job 的状态图"。源码里状态编码在 _state 字段中，下面用一张图展示所有合法转换路径：

tryMakeCompleting——状态原子切换的核心

这是协程库并发安全的心脏。所有状态变更都通过 CAS 循环在 _state 上完成。每次只允许一个线程成功执行状态切换：

// JobSupport.tryMakeCompleting 简化版
internal fun tryMakeCompleting(state: Any?, proposedUpdate: Any?): Any? {
    require(state !is Finishing) // 如果已完成, 抛异常
    return tryMakeCompletingSlowPath(state, proposedUpdate)
}

// 慢路径：CAS 循环直到成功或检测到竞争
// 返回 null → 切换成功, 需要继续做完成通知
// 返回非 null → 已有其他线程先完成, 返回当前状态
private fun tryMakeCompletingSlowPath(
    state: Incomplete, 
    proposedUpdate: Any?
): Any? {
    val list = getOrPromoteCancellingList(state) // 抓出子节点链表
    while (true) {
        if (state is Finishing) return state     // 已被其他线程完成
        val finalState = makeCompletion(proposedUpdate, list)
        if (_state.compareAndSet(state, finalState))
            return null                         // 成功！
        // CAS 失败 → 重新读取 _state, 继续循环
        state = _state.get()
    }
}

关键细节：① 所有并发控制只靠一个 AtomicReference+CAS——零锁、零 synchronized；② tryMakeCompleting 成功后返回 null，调用方负责后续通知（onCompletion 回调、父 Job 通知等）；③ CAS 失败意味着"你来晚了"——另一个线程或协程已经把状态改了。

invokeOnCompletion 与 CompletionHandler 链表

每个 Job 可以注册多个完成回调，它们以链表形式存在 _state 中。Job 完成时依次执行：

// 注册回调：CAS 把新 handler 插到链表头
invokeOnCompletion(onCancelling = false) { cause ->
    // Job 完成时调用
}

// 完成时通知所有 handler
private fun notifyCompletion(list: NodeList, cause: Throwable?) {
    var node = list.head
    while (node != null) {
        if (node is CompletionHandler) {
            node.invoke(cause) // 执行回调
        }
        node = node.next
    }
}

面试常问：为什么 invokeOnCompletion 能在 Job 已经完成后注册也能执行？因为注册时如果检测到 Job 已终态，立即同步调用回调，不会挂到链表上。

makeCompletingOnce——确保"只完成一次"的检查点

// 这是所有完成路径的统一入口
internal fun makeCompletingOnce(proposedUpdate: Any?): Any? {
    while (true) {
        val state = _state.get()
        if (state !is Incomplete) {
            // 已经完成了——返回现有完成状态
            return state
        }
        val finalState = tryMakeCompleting(state, proposedUpdate)
        if (finalState == COMPLETING_ALREADY) continue // 重试
        return finalState
    }
}

这一段暴露了一个关键面试点：协程完成时，先 CAS 到 Completing，然后等待所有子协程完成（子也一样递归等），最后才切到 Completed/Cancelled。所以 job.join() 不只是等直接子，而是在等整棵子树。

源码深潜：yield() 到底做了什么——不是 delay(0)

很多人把 yield() 当成"让出一下 CPU"，但源码里它和 Dispatcher 的交互比想象中复杂。

yield() 的完整调用链

// yield() 的定义——它是一个 suspend 函数
public suspend fun yield(): Unit = suspendCoroutineUninterceptedOrReturn { cont ->
    // 1. 获取当前协程的 Dispatcher
    val context = cont.context
    context.ensureActive()
    
    // 2. 通过 ContinuationInterceptor 包装 Continuation
    val intercepted = cont.intercepted()
    
    // 3. 调度的关键：dispatcher.dispatch(context, block)
    //    dispatch 把 block(Runnable) 放到线程的任务队列末尾
    //    如果同一个线程在循环里 yield，效果是"我排到队列末尾, 队列里其他人先执行"
    val dispatcher = context[ContinuationInterceptor] as? CoroutineDispatcher
        ?: return@suspendCoroutineUninterceptedOrReturn Unit
    
    // 4. 如果是 Unconfined Dispatcher，立即恢复（等价于无操作）
    if (dispatcher.isDispatchNeeded(context)) {
        dispatcher.dispatch(context, Runnable {
            intercepted.resumeWith(Result.success(Unit))
        })
    } else {
        // Unconfined / EventLoop 情况：直接继续
        intercepted.resumeWith(Result.success(Unit))
    }
    
    // 5. 返回 COROUTINE_SUSPENDED 表示"我挂起了, 后续通过 Continuation 恢复"
    COROUTINE_SUSPENDED
}

面试高频：AbstractCoroutine 的 handleJobException 做了什么？

// AbstractCoroutine 中的异常处理入口
protected open fun handleJobException(exception: Throwable): Boolean {
    // 1. 在 Context 中查找 CEH
    val handler = context[CoroutineExceptionHandler]
    if (handler != null) {
        handler.handleException(context, exception)
        return true  // 异常已被处理
    }
    // 2. 找不到 CEH → 交给全局处理器
    handleCoroutineExceptionImpl(context, exception)
    return false
}

关键：这个方法的返回值和 onCancelled 的 handled 参数不同。handleJobException 返回 true 表示"异常被 CEH 消费了，不要再传播"；而 onCancelled(cause, handled) 的 handled 表示"异常已经被处理过一轮"，即使用了 SupervisorJob 或 CEH。

工作窃取：findTask 三级查找策略

fun findTask(): Task? {
    // L1: 本地队列头部取(LIFO — CPU 缓存友好)
    localQueue.poll()?.let { return it }
    
    // L2: 全局阻塞队列
    globalBlockingQueue.removeFirstOrNull()?.let { return it }
    
    // L3: 随机偷其他 Worker 队列尾部(FIFO — 减少争用)
    return trySteal()
}

fun trySteal(): Task? {
    repeat(MAX_STEAL_ATTEMPTS) {
        val victim = workers[random()]
        if (victim !== this) {
            // 从尾部偷 — 双端队列的"另端"
            victim.localQueue.steal()?.let { return it }
        }
    }
    return null
}

fun dispatch(block: Runnable, taskContext: TaskContext) {
    if (taskContext.isBlocking) {
        incrementBlockingTasks()           // blockingTasks + 1
        globalBlockingQueue.addLast(task)   // 进全局队列
        signalBlockingWork()               // 可能需要更多线程!

    } else {
        // CPU 任务：优先进当前 Worker 的本地队列

    }
    signalCpuWork()  // unpark 一个空闲 Worker
}
面试加分：Default 和 IO 的区别不是"不同的线程池"，而是调度策略不同——CPU 走本地队列+工作窃取(最小化上下文切换)，IO 走全局队列+弹性扩容(避免阻塞占满CPU线程)。IO 默认上限 64，由 kotlinx.coroutines.io.parallelism 控制。

异常处理体系：launch、async、CEH 不能混着背

协程异常面试最爱问，因为很多错误代码能跑，但生命周期和错误边界全是隐患。

协程异常处理的难点，不在于有没有 try/catch，而在于你要先判断异常属于哪个边界：是当前业务可以恢复的错误，还是某个子任务失败导致整个父任务失败，还是已经逃逸到根协程的未捕获异常。不同边界对应的处理方式完全不同。

CoroutineExceptionHandler 经常被误用成“协程里的全局 catch”。实际上它更像最后的未捕获异常处理器，只会处理到达根协程、且没有被正常消费的异常。业务逻辑里能恢复的错误，比如接口返回失败、缓存读取失败、单个模块加载失败，仍然应该在业务边界用 try/catch、runCatching 或结果类型处理。

先区分 launch 和 async 的异常语义

launch 表达“启动一个不返回结果的任务”。如果它是根协程，未捕获异常会交给根的异常处理流程；如果它是子协程，异常会先影响父 Job。async 表达“启动一个未来会返回结果的任务”，异常会被保存在 Deferred 中，并在 await() 时重新抛出。

但这里有个常见误区：不能简单说“async 异常不会影响父协程”。如果 async 是普通父子结构里的子协程，它失败时自己的 Job 仍然失败，默认仍会取消父协程；只是调用者在 await() 时还会再次拿到这个异常。只有根 async 或 supervisor 边界里的 async，才更接近“异常等 await 消费”。

整体必须成功：coroutineScope 
suspend fun loadPage() = coroutineScope {
    val user = async { api.user() }
    val orders = async { api.orders() }

    Page(user.await(), orders.await())
}

任意一个子任务失败，整个页面加载失败，兄弟任务取消。这适合强依赖聚合。

允许部分失败：supervisorScope 
suspend fun loadHome() = supervisorScope {
    val feed = async { runCatching { api.feed() } }
    val ads = async { runCatching { api.ads() } }

    Home(feed.await().getOrDefault(emptyList()), ads.await().getOrNull())
}

每个模块自己处理错误，失败不会拖垮兄弟模块。这适合首页、聚合页和弱依赖模块。

CancellationException 不是错误，是协程控制信号

取消在协程里是一种正常完成路径。页面退出、用户刷新、新搜索关键词到来、父任务失败，都可能触发取消。框架用 CancellationException 传递这个信号，所以它默认不会被当成业务错误上报。

这也是为什么捕获 Throwable 或宽泛的 Exception 时要格外小心。如果你吞掉了 CancellationException，上层会以为任务已经正常处理完，取消传播就被截断了。很多“页面退出后请求还在跑”“旧搜索结果覆盖新结果”的问题，都和取消信号被误吞有关。

场景异常表现正确处理方式
根协程 launch未捕获异常会交给 CoroutineExceptionHandler 或线程未捕获异常处理器在根 Scope 配置 CEH 兜底
子协程 launch异常先取消父 Job，再由父向兄弟传播取消局部 try/catch 或使用 supervisor 边界
根协程 async异常保存在 Deferred 中，await() 时抛出；CEH 不兜这个异常对 await() try/catch
普通父子结构中的 async异常仍会使子 Job 失败，并默认取消父；await() 还会重新抛出需要隔离时放进 supervisorScope 或使用 SupervisorJob
CancellationException表示正常取消，默认不当作错误上报不要吞掉它；catch Throwable 时要重新抛出
 
// 错误：吞掉 CancellationException，会破坏取消
try {
    repository.sync()
} catch (t: Throwable) {
    logger.log(t)
}

// 正确：取消要继续向外传
try {
    repository.sync()
} catch (e: CancellationException) {
    throw e
} catch (t: Throwable) {
    logger.log(t)
}
面试表达模板：CoroutineExceptionHandler 只处理"未被消费的根协程异常"。它不是 try/catch 的替代品，也不会处理 async 已经封装到 Deferred 里的异常。业务可恢复错误应该在业务边界 catch，CEH 更像最后兜底日志。

// handleCoroutineException —— 根协程的异常处理器查找
public fun handleCoroutineException(context: CoroutineContext, exception: Throwable) {
    try {
        context[CoroutineExceptionHandler]?.let {      // 只在根的 Context 里找 CEH
            it.handleException(context, exception)      // 找到 → 调用 CEH
            return
        }
    } catch (t: Throwable) {}

}
为什么子协程上的 CEH 无效？异常沿 Job 树向上传播到达根节点时，handleCoroutineException 在根协程的 Context 中查找 CEH。子协程的 Context 不在查找范围内。所以 CEH 必须设在 Scope 根协程或 launch 的根协程 Context 上。

async 异常为什么 CEH 拿不到

async 创建的 DeferredCoroutine 覆写了 handleJobException，异常发生时不是调用 cancelImpl 传播，而是把异常存到内部 state 里，等 await() 时再重新抛出。所以异常不沿 Job 树向上传播，父 Job 不知道，CEH 也拿不到。

03 🌊 异步流 Flow · StateFlow · Channel

Flow：冷流、操作符链和 flowOn

Flow 是协程世界里的异步数据流。它的难点不是 API 名字，而是“谁触发生产、谁在哪个线程执行、异常从哪里被 catch”。

Flow 最适合用“管道”来理解，但这个管道有一个非常重要的特性：默认情况下，它不是提前运行的后台任务，而是一段等待被收集的声明。你写下 flow { emit(...) } 时，并没有立刻请求网络，也没有立刻发送数据。只有当某个协程调用 collect，上游 block 才开始执行。

这就是“冷流”的本质。冷流像一个函数，每次调用都会重新执行；热流像一个持续存在的状态源或事件源，是否有人订阅并不决定它是否存在。理解这一点，才能正确区分 Repository 返回 Flow、ViewModel 暴露 StateFlow、UI 收集状态这几层职责。

Flow 操作符不是列表转换，而是 collector 包装

很多人会把 Flow 的 map、filter 理解成集合操作符。语法上很像，但运行方式不同。集合是已经在内存里的数据；Flow 是一个异步发射过程。每个操作符会返回一个新的 Flow，它内部包装了下游 collector。等最终 collect 被调用时，这些包装才串成一条链。

所以数据方向和调用方向是相反的：collect 调用会从末端一路向上游建立链；emit 发生时，值再从上游一路经过 map、filter、catch、onEach 到达最终 collect。默认没有 buffer 时，上一个值要等下游处理完成，下一个值才会继续发射。

flowOn 为什么只影响上游

flowOn 不是“从这一行开始切线程”，而是“把我上游这段 Flow 放到指定上下文执行”。下游 collect 仍然运行在收集它的协程上下文里。这个设计让线程边界非常清晰：数据生产和转换可以在 IO 或 Default，UI 渲染仍然回到 Main。

当上下游上下文不同，Flow 通常需要在中间建立一个边界：上游在一个协程里发射，下游在另一个协程里接收，中间通过 channel 或类似机制传递值。这也是为什么 flowOn 会影响取消、异常和缓冲行为，不能把它当成普通函数调用里的 withContext。

读后判断：看到一条 Flow 链时，先标出三个边界：冷流从哪里开始执行，flowOn 影响到哪里，catch 能捕获哪一段。大多数 Flow bug 都能从这三件事里定位。

独立于收集者存在，多个收集者共享同一份数据。典型代表是 StateFlow 和 SharedFlow。

private class SafeFlow<T>(private val block: suspend FlowCollector<T>.() -> Unit) 
    : AbstractFlow<T>() {

        collector.block()   // 你的 emit(1); emit(2) 在这里执行
    }
}

// collect 方法最终调用

    collectSafely(safeCollector)   // SafeCollector 做了上下文一致性检查
}

        // StateFlow 自带去重: 新值和旧值 equals 相同时不通知
    }

Flow 进阶：flatMap 三兄弟、背压策略与 channelFlow

这是 Flow 面试最能拉开差距的考点——flatMapConcat/Merge/Latest 的并发语义、buffer/conflate 的背压策略、以及 channelFlow 什么时候该替代 flow{}。

flatMap 三兄弟：同一个操作符名，三种截然不同的并发语义

flatMapConcat 
flowOf("A", "B", "C")
    .flatMapConcat { requestFlow(it) }
    .collect { println(it) }
// 输出：A1, A2, B1, B2, C1, C2
// 完全串行：A 的流走完才走 B

语义：顺序拼接。每个上游值启动的内部流，必须完全消费完才会消费下一个。适合"上一批结果没回来之前不能开始下一批"的场景。

flatMapMerge 
flowOf("A", "B", "C")
    .flatMapMerge { requestFlow(it) }
    .collect { println(it) }
// 输出可能是：A1, B1, C1, A2, B2, C2
// 完全并发：A/B/C 的内部流同时执行

语义：并发融合。所有上游值启动的内部流同时执行，下游收到的是各流输出按时间交错的混合。默认 concurrency=16，可通过参数限制。

flatMapLatest 
flowOf("A", "B", "C")
    .flatMapLatest { requestFlow(it) }
    .collect { println(it) }
// 输出：只有 C1, C2
// 新值到来：取消旧内部流，启动新内部流

语义：只关心最新。上游发射新值时，取消当前正在执行的内部流，启动新流。这是搜索框防抖的标准模式。

三兄弟并发模型一览

操作符内部流并发旧流行为典型场景
flatMapConcat否 — 严格串行等旧流完成后处理下一个依次加载分页数据
flatMapMerge是 — 可设 concurrency全部保留，并发执行同时请求多个 API, 谁回来收谁
flatMapLatest是 — 但旧流被取消新值到达时取消旧流搜索建议、切换 tab 加载

buffer 与 conflate：当生产者快于消费者

// 场景：生产 100 个值，消费者每个处理要 100ms
flow {
    repeat(100) { emit(it) }
}.collect { 
    delay(100)  // 消费慢
}

// 默认：生产者挂起等待消费者完成 → 100 个值耗时 100×100ms = 10s

// buffer()：生产者和消费者在不同协程并发
flow { repeat(100) { emit(it) } }
    .buffer(10)  // 缓冲区 10 个，满了生产者挂起
    .collect { delay(100) }
// 效果：生产者和消费者并行，整体时间接近 max(生产, 消费)

// conflate()：缓冲区大小=1，新值覆盖旧值
flow { repeat(100) { emit(it) } }
    .conflate()  // 每次只保留最新未消费的值
    .collect { delay(100) }
// 效果：消费者可能收到 0, 2, 5, 8, 15, ...（跳帧式的值）
策略缓冲区满时行为丢值?适用场景
默认（无 buffer）0生产者挂起等消费者否简单场景
buffer(n)n生产者挂起等缓冲区有空间否生产消费速度不匹配
conflate()1新值覆盖旧值，生产者不挂起是只关心最新值（如传感器数据）
buffer(Channel.UNLIMITED)无限生产者永不挂起否（但 OOM 风险）生产者不应被消费者阻塞

channelFlow vs flow {}：什么时候该用 channelFlow

面试高频题——"channelFlow 和 flow {} 的区别是什么？"答案不是"channelFlow 可以并发"，而是它们有不同的并发模型和上下文行为：

// flow {}：你的 block 是 suspend 的——串行，一个线程
flow {
    emit(api.users())       // 挂起
    emit(api.orders())      // 等上面的回来了再继续
}
// emit 是挂起函数，在 block 线程内完成

// channelFlow {}：你的 block 有 ProducerScope——可以并发发射
channelFlow {
    launch {                    // ← 新协程！
        send(api.users())       // send 不是 suspend，是挂起
    }
    launch {                    // ← 又一个新协程！
        send(api.orders())
    }
    awaitClose { /* 清理 */ }  // channelFlow 特有的关闭回调
}
flow {} 的行为
  • block 是 suspend，在 单个协程内执行
  • emit 是挂起函数，受上下文保持约束
  • 不能在里面 launch
  • 适合"声明式数据描述"
channelFlow {} 的行为
  • block 有 ProducerScope，可以 launch
  • 通过底层 Channel 传输数据
  • send 是挂起函数，可跨协程
  • 不受上下文保持约束——可以在 block 里自由切线程
  • 需要 awaitClose 处理取消
选型建议：能描述数据生产逻辑用 flow {}；需要内部并发、上下文切换、监听回调转 Flow（callbackFlow 是它的简化版）用 channelFlow {}。大多数项目代码 90% 的 Flow 都是 flow {} 就够。

StateFlow / SharedFlow：UI 状态和一次性事件分开建模

Android 页面里最常见的错误，是把状态和事件都塞进一个流里。状态要可重放、可读当前值；事件通常只想消费一次。

在 Android 页面中，状态和事件看起来都像“要通知 UI 的东西”，但它们的语义完全不同。状态描述的是“当前页面是什么样子”，例如是否加载中、列表数据是什么、错误提示文案是什么。事件描述的是“发生了一次动作”，例如跳转、弹 Toast、显示 Snackbar、请求权限。

StateFlow 适合状态，因为它总有一个当前值，新订阅者一上来就能拿到最新状态。旋转屏幕、页面重建、Compose 重组时，这个特性非常有用：UI 不需要等下一次网络回调，就能根据当前状态恢复界面。

但正因为 StateFlow 会保留最新值，它不适合表达一次性事件。如果你把 NavigateToDetail 放进 StateFlow，新 UI 订阅后可能再次收到这个旧值，导致重复跳转。很多项目用“消费后把事件置空”来补救，但这会把事件生命周期塞回状态对象里，复杂页面很快变得混乱。

状态：可以被重复观察 
data class LoginState(
    val loading: Boolean = false,
    val account: String = "",
    val error: String? = null
)

无论 UI 什么时候订阅，都应该能得到当前状态。旧状态被新 UI 重新渲染是合理的。

事件：只应该被消费一次 
sealed interface LoginEvent {
    data object NavigateHome : LoginEvent
    data class Toast(val message: String) : LoginEvent
}

导航和 Toast 是一次性动作。新 UI 不应该因为重新订阅就重复执行旧事件。

StateFlow 的 equals 去重既是优点也是陷阱

MutableStateFlow.value 更新时，如果新旧值按 equals 判断相等，收集者不会收到新值。这和不可变 data class 很搭：字段不变就不触发无意义刷新。但如果你在原地修改可变 List，再把同一个对象塞回 StateFlow，可能因为引用或 equals 语义导致 UI 不更新。

更稳的写法是让 UI 状态不可变，更新时用 copy 创建新对象，集合也尽量使用新列表。这样状态变化清晰，Compose 重组、DiffUtil、日志追踪都会更可控。

类型特点适合场景注意点
StateFlow必须有初始值，只保留最新值，按 equals 去重页面 UI 状态、表单状态、加载状态用不可变 data class，更新用 copy
SharedFlow可配置 replay、buffer、溢出策略导航、Toast、Snackbar、埋点事件一次性事件通常 replay = 0
Channel点对点通信，receive 消费后移除严格单消费者任务队列配置容量和关闭语义要谨慎
 
data class ProfileUiState(
    val loading: Boolean = false,
    val user: User? = null,
    val error: String? = null
)

class ProfileViewModel(
    private val repo: UserRepository
) : ViewModel() {
    private val _state = MutableStateFlow(ProfileUiState())
    val state: StateFlow<ProfileUiState> = _state.asStateFlow()

    private val _events = MutableSharedFlow<UiEvent>()
    val events: SharedFlow<UiEvent> = _events.asSharedFlow()

    fun load() = viewModelScope.launch {

            .onFailure { e ->

Compose 侧建议：使用生命周期感知的收集方式，例如 collectAsStateWithLifecycle()。传统 View 侧使用 repeatOnLifecycle，避免页面不可见时仍然收集 UI 数据。

Channel：协程之间的通信管道

Flow 是"一个生产者对多个消费者的数据流"，Channel 是"协程到协程的点对点消息传递"。它弥补了 launch/async 无法在运行中的协程之间传递数据的问题。

Channel 解决什么问题？

用 launch 启动多个协程后，如果生产者协程产出了数据需要实时发给消费者协程，该怎么办？全局变量不安全，Flow 适合声明式管道但不适合双向通信，async/await 只返回一个结果。Channel 就是为这个场景设计的——一个协程 send，另一个协程 receive，线程安全，自带挂起/恢复。

生产者-消费者模式 
val channel = Channel<Int>(capacity = Channel.UNLIMITED)

launch(Dispatchers.Default) {    // 生产者
    for (i in 1..5) {
        channel.send(i)
        delay(100)
    }
    channel.close()               // 生产完毕后关闭
}

launch(Dispatchers.IO) {         // 消费者
    for (value in channel) {     // 等价于 while(receive) 
        process(value)
    }
}

send() 在 Channel 满时挂起；receive() 在 Channel 空时挂起。channel.close() 后消费者遍历自动结束。

四种 Channel 类型的缓冲区策略

类型缓冲区send 何时挂起经典场景
RENDEZVOUS0（默认）必须等消费者来 receive严格同步、背压精确
BUFFERED64缓冲区满才挂起吞吐优先、允许少量开销
CONFLATED1（合并）新值覆盖旧值只关心最新值（如 UI 更新）
UNLIMITED无限永不挂起消费者一定比生产者快的场景

fan-out 和 fan-in：多对多通信

Channel 可以被多个消费者共享（fan-out），也可以由多个生产者写入（fan-in），但每个消息只会被一个消费者接收（非广播）。如果需要广播，用 BroadcastChannel（已废弃，推荐 SharedFlow）。

// Fan-out: 多个消费者竞争同一个 Channel
val channel = Channel<Int>(BUFFERED)
repeat(5) { 
    launch(Dispatchers.Default) {
        for (msg in channel) { process(it, msg) }
    }
}
launch { repeat(10) { channel.send(it) }; channel.close() }
// 10 个消息被 5 个消费者竞争处理

Channel 与 Flow 如何桥接

// Channel → Flow
val flow = channel.receiveAsFlow()

// Flow → Channel
flow.produceIn(scope)  // 返回 ReceiveChannel

receiveAsFlow() 把 Channel 包装成 Flow，多个收集者可以同时消费（实际上被 Channel 的消费语义限制，通常只单消费）。produceIn 反向桥接，让 Flow 的值灌入 Channel。

源码摘要

// Channel 接口极其简洁
public interface Channel<E> : SendChannel<E>, ReceiveChannel<E> {
    // SendChannel: send(element) / trySend / close
    // ReceiveChannel: receive() / tryReceive / iterator
    val capacity: Int
}
性能提示：Channel 在协程间传递数据比 Flow 更直接（少了声明式操作符链的开销），但可以维护的状态更少。一般规则：单纯的协程到协程消息用 Channel，多消费者需要背压/操作符/变换的用 Flow。

高级模式：Ticker 定时通道

// ticker: 每隔一段时间自动发送 Unit 的 Channel
val ticker = ticker(delayMillis = 1000, initialDelayMillis = 0)
var count = 0
for (tick in ticker) {
    if (++count > 5) { ticker.cancel(); break }

}

// Tick 2 at 1000ms, Tick 3 at 2000ms, ...

高级模式：produce 构建器

// produce: 创建一个协程 + Channel 的组合，协程负责生产数据
fun CoroutineScope.produceUsers(): ReceiveChannel<User> = produce {
    while (isActive) {
        send(fetchNextUser())
        delay(500)
    }
}
// produce 协程取消时自动关闭 Channel，消费者循环自动退出

Channel 关闭的信号机制

// Channel 关闭后，for 循环自动终止
for (value in channel) { ... }  // close() 后循环结束

// receive() 在关闭后抛 ClosedReceiveChannelException
// receiveCatching() 返回 Success/Failure 不抛异常
val result: ChannelResult<T> = channel.receiveCatching()
result.onSuccess {  }   // 正常值
      .onFailure {  }   // Channel 已关闭

Android 落地架构：让协程待在正确的层

稳定的协程代码不是“哪里都 launch 一下”，而是每一层只承担自己的职责。

Android 项目里，协程最容易被滥用的地方是 Scope 边界。很多问题不是 API 用错，而是协程启动在了错误的层：Repository 持有页面 Scope，Fragment 里直接开长期任务，ViewModel 暴露可变 StateFlow，UI 不按生命周期收集 Flow。代码短期能跑，长期就会出现泄漏、重复请求、状态错乱。

一个清晰的分层原则是：UI 层负责展示和用户输入，ViewModel 层负责把用户意图转换成状态变化，Data 层负责提供数据能力。Data 层不应该知道页面什么时候销毁，UI 层也不应该知道网络请求该切到哪个 Dispatcher。

Repository 暴露能力，不持有页面生命周期

Repository 更适合暴露 suspend 函数或 Flow。一次性请求用 suspend，持续数据源用 Flow。它可以在内部切到 IO，也可以组合缓存和网络，但不应该直接使用 viewModelScope。谁调用它，谁决定这个任务属于哪个生命周期。

ViewModel 启动页面任务，并把结果变成 UI 状态

ViewModel 是页面协程最常见的启动点。它用 viewModelScope 承接页面生命周期，用私有 MutableStateFlow 管状态，对外只暴露只读 StateFlow。异常在这里被转换成错误状态或一次性事件，而不是直接把 Throwable 扔给 UI。

UI 按生命周期收集，不保存业务协程

Fragment 或 Activity 收集 Flow 时，应该跟随可见生命周期启动和取消。传统 View 使用 repeatOnLifecycle，Compose 使用生命周期感知的收集 API。这样页面不可见时，UI 收集会停止，上游也有机会因为没有订阅者而停下来。

UI 层

  • 只收集状态、触发用户意图。
  • 不要在 Activity/Fragment 里长期持有裸 Scope。
  • 按生命周期收集 Flow。

ViewModel 层

  • 用 viewModelScope 启动页面任务。
  • 暴露不可变 StateFlow。
  • 把异常转成 UI 状态或事件。

Data 层

  • 优先暴露 suspend 或 Flow。
  • 内部切到 IO 或 Default。
  • 不要依赖 Android UI 生命周期。
// Fragment / View
viewLifecycleOwner.lifecycleScope.launch {
    repeatOnLifecycle(Lifecycle.State.STARTED) {
        viewModel.state.collect { state ->
            render(state)
        }
    }
}
为什么要 repeatOnLifecycle：它会在生命周期进入目标状态时启动收集，离开时取消收集。这样 UI 不可见时，上游能按取消语义停下来，减少泄漏和无意义刷新。

一套可复用的加载模式

fun refresh() = viewModelScope.launch {
    _state.update { it.copy(refreshing = true, error = null) }
    try {
        val data = repository.refresh()
        _state.update { it.copy(refreshing = false, items = data) }
    } catch (e: CancellationException) {
        throw e
    } catch (e: IOException) {
        _state.update { it.copy(refreshing = false, error = "网络异常") }
    } catch (e: Throwable) {
        _state.update { it.copy(refreshing = false, error = "未知错误") }
    }
}

场景实验室：把协程放回真实业务

真正能拉开水平差距的不是会写 launch，而是面对页面加载、搜索、部分失败、退出取消、事件消费时，能选对结构并解释原因。

下面的场景不是为了展示“某个 API 的标准写法”，而是训练判断能力。同样是并发请求，有的场景要整体失败，有的场景要部分可用；同样是 Flow，有的场景要保留所有值，有的场景只要最新值。API 选择取决于业务语义，而不是哪个写法看起来更高级。

读每个案例时建议先问三件事：第一，这些任务之间是强依赖还是弱依赖；第二，旧任务是否应该被新任务取消；第三，结果应该进入持续状态还是一次性事件。能回答这三件事，协程结构基本就不会选错。

Case 1

首页多个模块并发加载，部分失败不拖垮全页

业务诉求：用户信息、推荐流、广告、未读数可以并发加载。推荐流失败不应该让用户信息也消失。

suspend fun loadHome(): HomeUiModel = supervisorScope {
    val profile = async { runCatching { repo.profile() } }
    val feed = async { runCatching { repo.feed() } }
    val badges = async { runCatching { repo.badges() } }

    HomeUiModel(
        profile = profile.await().getOrNull(),
        feed = feed.await().getOrElse { emptyList() },
        badges = badges.await().getOrDefault(0)
    )
}

判断标准：多个模块互不强依赖，用 supervisorScope 做故障隔离；如果任何模块失败都必须整体失败，才用 coroutineScope。

Case 2

搜索框防抖，旧请求自动取消

业务诉求：用户持续输入时，不应该每个字符都请求；新关键词到来后，旧关键词的网络结果不应该覆盖新结果。

val results = queryFlow
    .debounce(300)
    .map { it.trim() }
    .distinctUntilChanged()
    .flatMapLatest { keyword ->
        if (keyword.isBlank()) flowOf(emptyList())
        else repository.search(keyword)
    }
    .stateIn(viewModelScope, SharingStarted.WhileSubscribed(5_000), emptyList())

判断标准：flatMapLatest 表达“只要最新”，旧上游会被取消；如果要保留所有请求顺序，才考虑 flatMapConcat。

Case 3

页面退出后，网络请求必须取消

业务诉求：详情页退出后，请求不应该继续更新已销毁页面；ViewModel 清理时任务要一起结束。

class DetailViewModel(
    private val repo: DetailRepository
) : ViewModel() {
    fun load(id: Long) = viewModelScope.launch {
        _state.update { it.copy(loading = true) }
        val detail = repo.detail(id)
        _state.update { it.copy(loading = false, detail = detail) }
    }
}

判断标准：页面任务用 viewModelScope，UI 收集用生命周期感知 API。不要把页面请求丢进 GlobalScope。

测试与调试：控制 Dispatcher 和虚拟时间

协程测试的关键是可控：不要让测试依赖真实线程、真实 delay、真实主线程。

协程测试最怕“偶尔过、偶尔不过”。如果测试依赖真实线程调度、真实 delay、真实主线程，就会变得慢且不稳定。好的协程代码从设计时就应该考虑可测试性：Dispatcher 可替换，时间可推进，状态可观察，异常路径可断言。

runTest 的价值在于提供一个测试用的协程环境和虚拟时间。代码里写 delay(5_000)，测试不需要真的等 5 秒；你可以用测试调度器推进时间，或者用 advanceUntilIdle() 让当前队列中的协程任务跑完。这样测试既快，又能稳定复现执行顺序。

为什么硬编码 Dispatchers.IO 会让测试难写

如果 Repository 内部直接写死 Dispatchers.IO，测试时这段代码会跑到真实 IO 线程池。你无法精确控制它什么时候执行，也很难让虚拟时间接管它。更好的方式是把 Dispatcher 当作依赖注入：生产环境传 Dispatchers.IO，测试环境传 StandardTestDispatcher。

难测试：调度器写死 
class UserRepo(private val api: Api) {
    suspend fun load() = withContext(Dispatchers.IO) {
        api.load()
    }
}

测试无法接管 IO 线程池，执行时机不稳定。

可测试：调度器注入 
class UserRepo(
    private val api: Api,
    private val io: CoroutineDispatcher
) {
    suspend fun load() = withContext(io) { api.load() }
}

测试可以传入 TestDispatcher，用虚拟时间和任务队列控制执行。

测试 ViewModel 时，应该断言状态流转

ViewModel 测试不应该只断言“调用了某个方法”。更有价值的是断言 UI 状态如何变化：初始状态是什么，触发加载后是否进入 loading，成功后是否有数据，失败后是否出现错误状态，取消时是否没有错误提示。这样测试覆盖的是用户真正能感知的行为。

测试 Flow 时，也不要只测第一个值。很多 bug 出现在第二次、第三次发射：防抖是否生效、旧请求是否被 flatMapLatest 取消、stateIn 是否重复订阅上游、错误是否被转换成空列表或错误状态。这些都需要按时间推进和多次收集来验证。

性能与反模式：协程不是万能加速器

协程让异步代码更好组织，但不会让阻塞代码自动变快。错误使用 Dispatcher、Flow 和 Scope，反而会制造卡顿、泄漏和难以复现的异常。

协程经常被误解成“轻量级线程”，于是有人以为多开协程就能提升性能。实际上，协程降低的是异步代码的组织成本和等待成本，不会让 CPU 计算变少，也不会让阻塞 IO 自动变快。如果底层 API 本身阻塞，协程只是帮你把阻塞放到更合适的线程池里；如果算法本身很重，协程也不能替你省掉计算。

排查协程性能问题，可以从四个方向入手：线程是否被阻塞，取消是否及时生效，Flow 是否产生过多中间值，生命周期是否导致重复收集或重复请求。很多页面卡顿并不是某个函数特别慢，而是这些小问题叠加起来：输入框每个字符都请求、旧请求没取消、列表每次发射都全量刷新、页面不可见时仍然收集。

反模式 1：用协程包装阻塞，却没有切 Dispatcher

suspend 本身不保证非阻塞。如果你在 suspend 函数里直接调用阻塞式 SDK，而调用方又在 Main 上执行，就仍然会卡主线程。正确做法是在数据层明确切到 IO，或者使用真正非阻塞的挂起 API。

反模式 2：Flow 发射太频繁，下游处理太重

Flow 默认是顺序的，但这不代表它天然高效。搜索输入、滚动位置、传感器数据、数据库变更都可能高频发射。如果每个值都触发网络请求、复杂 diff 或全页面状态重建，就会造成抖动。此时应该根据语义选择 debounce、sample、distinctUntilChanged、conflate 或 flatMapLatest。

反模式 3：共享上游没有共享，导致重复请求

冷流每次 collect 都会重新执行。如果两个 UI 区域分别 collect 同一个网络 Flow，可能触发两次请求。对于昂贵上游，应在 ViewModel 中用 stateIn 或 shareIn 共享结果，并用 SharingStarted.WhileSubscribed 控制没人订阅时是否停掉上游。

反模式清单

  • GlobalScope.launch 启动业务任务，生命周期无人管理。
  • 在 Main 线程执行阻塞 IO 或大量 JSON 解析。
  • 用 runBlocking 包住 Android UI 代码。
  • catch Throwable 后吞掉 CancellationException。
  • 把一次性事件建模成 StateFlow，旋转屏幕后重复消费。
  • 对高频 Flow 每个值都刷新整个页面。

  • 多个独立请求用 async 并发，但用结构化边界等待。
  • 共享昂贵上游用 stateIn 或 shareIn。
  • 长循环定期 yield 或 ensureActive。

collectAsStateWithLifecycle——正确收集 StateFlow 的标准姿势

这是 Compose 里最重要的一个 API，也是面试高频点。它做了两件事：① 把 Flow 转成 Compose State；② 在生命周期低于 STARTED 时自动停止收集。

// 需要依赖：androidx.lifecycle:lifecycle-runtime-compose
@Composable
fun UserScreen(viewModel: UserViewModel) {
    // ✅ 正确：页面不可见时停止收集，回来时重新订阅
    val uiState by viewModel.state.collectAsStateWithLifecycle()
    
    // ❌ 错误：即使页面不可见也在持续收集，浪费 CPU + 可能泄漏
    // val uiState by viewModel.state.collectAsState()
    
    when (val state = uiState) {
        is Loading -> CircularProgressIndicator()
        is Success -> UserList(state.users)
        is Error -> ErrorMessage(state.message)
    }
}
关键区别：collectAsState() 只在离开组合时停止。collectAsStateWithLifecycle() 在页面不可见时（STARTED 以下）就停止。前者适合生命周期无关的 Dialog 级别状态，后者适合页面级状态。

LaunchedEffect——在组合生命周期内执行挂起代码

它接受一个 key，当 key 变化时取消旧协程并启动新协程。可以理解成"组合感知的 launch"：

@Composable
fun UserDetail(userId: String, repository: UserRepository) {
    var user by remember { mutableStateOf(null) }
    var error by remember { mutableStateOf(null) }
    
    // userId 变化时：取消旧请求，发起新请求
    LaunchedEffect(userId) {
        user = null
        error = null
        try {
            user = repository.loadUser(userId)
        } catch (e: Exception) {
            error = e.message
        }
    }
    
    user?.let { UserView(it) }
    error?.let { ErrorBanner(it) }
}

面试坑：LaunchedEffect 默认使用 Main Dispatcher。如果你在里面做耗时操作但不切线程，会卡 UI。正确写法是 LaunchedEffect(key) { withContext(Dispatchers.Default) { ... } }。

rememberCoroutineScope——用户事件驱动的协程启动点

和 LaunchedEffect 的区别：rememberCoroutineScope 不自动启动协程，它给你一个绑定组合生命周期的 Scope，供按钮点击等用户事件使用：

@Composable
fun RefreshableList(viewModel: ListViewModel) {
    val scope = rememberCoroutineScope()
    val state by viewModel.state.collectAsStateWithLifecycle()
    
    Button(onClick = {
        scope.launch {          // ← 用户点击时启动协程
            viewModel.refresh()
        }
    }) {
        Text("刷新")
    }
    // 离开组合时 scope 自动 cancel，无需手动管理
}

produceState——异步数据转 Compose State

把任意挂起函数的结果变成 Compose State，适合"一个值，来源是异步的"场景：

@Composable
fun loadUser(userId: String): State = produceState(
    initialValue = null,
    key1 = userId
) {
    // 这个 block 是 suspend 的，在 LaunchedEffect 里执行
    value = withContext(Dispatchers.IO) {
        api.loadUser(userId)
    }
}

snapshotFlow——Compose State 变 Flow（反向桥接）

这是最少用但最容易被问到的 API：把 Compose 的 Snapshot State 转成 Flow。典型场景是"当某个 UI 状态变化时触发协程逻辑"：

@Composable
fun SearchBar(viewModel: SearchViewModel) {
    var query by remember { mutableStateOf("") }
    
    LaunchedEffect(Unit) {
        snapshotFlow { query }
            .debounce(300)      // 防抖
            .filter { it.length > 2 }
            .collect { viewModel.search(it) }
    }
    
    TextField(value = query, onValueChange = { query = it })
}

协程"轻量"的本质——三条技术原因

原因 1：无新线程创建

协程是语言级抽象，运行在已有线程上。10,000 个协程 ≠ 10,000 个线程。Dispatchers.Default 始终只有 CPU 核心数个线程。

原因 2：挂起释放线程

挂起时协程对象离开调用栈，线程可以去执行其他协程。这与线程阻塞完全不同——阻塞会占住线程栈和寄存器，什么都做不了。

原因 3：无系统调用

协程的创建、切换、取消都在用户态完成，不需要陷入内核。唯一的系统调用是线程 park/unpark（仅在 Worker 空闲/唤醒时触发）。

但不代表零成本

每个协程仍需要分配 Job + Continuation + StateMachine 对象。大量协程同时活跃会增加 GC 压力。所以"百万协程"是可能的，但不代表应该无节制开协程。

// 协程轻量但非零成本的典型场景
launch(Dispatchers.Default) {
    // 这个协程对象分配了 ~200 bytes
    delay(1000)  // 挂起，释放 Worker 线程
    // Worker 线程这 1 秒内可以执行其他协程的任务
}
面试表达模板："协程的轻量体现在三个层面：创建时不需要新线程和内核栈分配（内存开销 ~1000x 差距）；运行时通过用户态调度避免内核上下文切换（切换成本 ~100x 差距）；挂起时释放线程而非阻塞（释放的是线程资源，不是计算资源）。但轻量不是零成本——每个协程仍有对象分配和 GC 压力。"

Select 表达式：谁先完成用谁

这是一个被低估的协程 API——它能让你同时监听多个 Channel 或 Deferred，选择最先完成的那个，并取消未完成的。

select 的核心语义

假设你同时发起两个网络请求，但只需要最快返回的那个结果。传统写法需要额外的变量、Job 管理、手动取消另一个。select 在语言层面解决了这个问题：

suspend fun fetchFastest(): Result = coroutineScope {
    val deferred1 = async { api.sourceA() }
    val deferred2 = async { api.sourceB() }
    
    select<Result> {
        deferred1.onAwait { it }      // 如果 deferred1 先返回 → 选它
        deferred2.onAwait { it }      // 如果 deferred2 先返回 → 选它
    }.also {
        // select 返回后，未被选中的 Deferred 自动被取消
    }
}

select 支持的操作

操作作用示例
onAwait {}等待 Deferred 完成deferred.onAwait { it }
onReceive {}等待 Channel receivechannel.onReceive { it }
onSend(value) {}等待 Channel send 成功channel.onSend(data) { }

select 的实际应用场景

// 场景1: 超时竞赛 —— 两个服务同时查，用快的
val result = select {
    async { cache.get(id) }.onAwait { "缓存命中: $it" }
    async { network.fetch(id) }.onAwait { "网络命中: $it" }
    onTimeout(200) { "两者都太慢，给兜底" }
}

// 场景2: 多 Channel 消费 —— 同时监听多个事件源
select {
    userEvents.onReceive { handleUserEvent(it) }
    systemEvents.onReceive { handleSystemEvent(it) }
    ticker.onReceive { handleTick() }
}
select 是实验性 API，需要 @OptIn(ExperimentalCoroutinesApi::class)。它的内部实现用到协程底层的一次性 Continuation，选择一个分支后，其他分支的 Continuation 被取消。

select 与 Channel 的经典配合：扇出消费

// 多个 Channel 汇聚到同一个 select
suspend fun consumeMulti(ch1: ReceiveChannel<Int>, ch2: ReceiveChannel<String>) {
    while (!ch1.isClosedForReceive || !ch2.isClosedForReceive) {
        select<Unit> {
            ch1.onReceive { println("Channel1: $it") }
            ch2.onReceive { println("Channel2: $it") }
        }
    }
}

CoroutineStart 四种模式：控制协程的启动时机

99% 的场景用默认的 DEFAULT 就够了，但当你需要"懒加载协程"或"在启动的线程先执行一段"时，另外几种 Start 模式是关键。

模式行为使用场景
DEFAULT立即调度到 Dispatcher99% 的场景
LAZY创建但不执行，等 start()/join()/await()需要显式控制启动时机
ATOMIC立即执行，但在第一个挂起点前不可取消启动后必须至少执行到第一个挂起点
UNDISPATCHED在当前线程直接执行到第一个挂起点，然后调度优化：减少一次线程切换
 
// LAZY: 显式控制启动，start() 后可多次调用
val job = launch(start = CoroutineStart.LAZY) {
    expensiveInit()
}
// 此时协程还没开始
job.start()  // 现在才开始

// ATOMIC: 第一个挂起点前不可取消
val job = launch(start = CoroutineStart.ATOMIC) {
    criticalSetup()    // cancel() 到这之前都无效
    delay(100)         // 第一个挂起点，这里开始可以取消
    restOfWork()
}

// UNDISPATCHED: 减少一次线程切换
launch(Dispatchers.Default, start = CoroutineStart.UNDISPATCHED) {
    // 第一段代码在当前线程（可能是 Main）立即执行
    // 遇到第一个挂起点后再切到 Default 线程池
}
UNDISPATCHED 陷阱：它在当前线程立即执行到第一个挂起点，这意味着如果你的调用线程是 Main 线程且 block 里有耗时操作，会卡 UI。慎用。

进阶 API：面试加分项和日常效率工具

这节覆盖了 launch/async 之外那些"知道的人不多但非常好用"的协程 API。每一个都是面试里能区分 70 分和 90 分的知识点。

runBlocking：阻塞世界的桥梁

runBlocking 的名字就是一个警告：它真的会阻塞当前线程，直到内部协程执行完毕。它存在的唯一理由是在主函数、测试或初始化阶段——这些地方没有现成的 Scope。

// main() 里运行协程的唯一方式，直到 Kotlin 原生支持 suspend main
fun main() = runBlocking {
    launch { delay(1000); println("Hello") }
}

// 测试里不需要，用 runTest 更好
// 初始化时如果实在拿不到 Scope，也尽量用 runBlocking 的最小作用域
runBlocking {
    val data = async { loadInitialData() }
    initialize(data.await())
}
面试踩分点：说出 runBlocking 会阻塞调用线程、在 suspend 函数内部绝不应该使用、它是"把异步世界桥接到同步世界的最后手段"。

withTimeout：给协程加一个死线

// withTimeout — 超时抛 TimeoutCancellationException
try {
    val result = withTimeout(3000) { api.fetchData() }
} catch (e: TimeoutCancellationException) {
    showError("请求超时")
}

// withTimeoutOrNull — 超时返回 null（更常用）
val result = withTimeoutOrNull(3000) { api.fetchData() } ?: cachedData

内部原理：withTimeout 启动一个计时协程，超时后调用 job.cancel(TimeoutCancellationException)。因为是 CancellationException，不会触发 CEH——但你的 finally 块会正常执行清理。

yield：主动让出执行权

// 在 CPU 密集循环中给其他协程执行机会
withContext(Dispatchers.Default) {
    repeat(10000) {
        heavyComputation(it)
        if (it % 100 == 0) yield()     // 每 100 次让出一次
    }
}
// yield = suspend + 立即恢复，效果是把自己排到线程队列末尾

yield 不产生任何数据变化，它只是"我先排队，让你先跑"。和 delay(0) 的区别：delay 调用的是调度器的时间队列，yield 是线程池的任务队列，在单线程 Dispatcher 上效果相同。

Mutex：协程安全的互斥锁

多协程同时修改共享资源时，不能用 Java 的 synchronized 或 ReentrantLock——它们会阻塞线程，破坏协程的挂起语义。

val mutex = Mutex()
var counter = 0

repeat(1000) {
    launch(Dispatchers.Default) {
        mutex.withLock {
            counter++    // 同一时刻只有一个协程能进入这里
        }
    }
}
// withLock 是挂起函数，等待时不占用线程
和 synchronized 的区别：synchronized 阻塞线程，mutex.withLock 挂起协程。所以在 Main 线程用 mutex 不会卡 UI（等待时线程可以干别的），而 synchronized 会 ANR。

callbackFlow：把回调 API 变成 Flow

这是把传统回调库（LocationManager、BroadcastReceiver、SensorManager）接入协程世界的最强工具：

// 传统回调 → Flow 的标准模板
fun LocationManager.locationFlow(): Flow<Location> = callbackFlow {
    val listener = LocationListener { location ->
        trySend(location)         // 把回调数据发送到 Channel
    }
    requestLocationUpdates(provider, 0, 0f, listener)
    
    awaitClose {                  // Flow 收集取消时自动执行清理
        removeUpdates(listener)
    }
}
// 收集者取消时，awaitClose 自动触发，防止内存泄漏

shareIn / stateIn：冷流转热流

你写了一个 flow { } 构建的冷流，现在多个 UI 组件都想订阅同一个数据源。每个组件 collect 会导致重复网络请求——你需要把它变成热流：

// stateIn: 冷流 → StateFlow
val userState: StateFlow<User?> = repository.userFlow()
    .stateIn(
        scope = viewModelScope,
        started = SharingStarted.WhileSubscribed(5000),
        initialValue = null
    )

// shareIn: 冷流 → SharedFlow
val events: SharedFlow<Event> = eventFlow
    .shareIn(
        scope = viewModelScope,
        started = SharingStarted.WhileSubscribed(),
        replay = 0
    )

// SharingStarted 三种策略:
// Eagerly:    立即启动，永不停止
// Lazily:     第一个订阅者出现时启动，永不停止
// WhileSubscribed: 有订阅者时活跃，最后订阅者消失后等 N ms 关闭
标准选择：需要持有当前状态 → stateIn + WhileSubscribed(5000)；需要共享一次性事件 → shareIn + WhileSubscribed() + replay=0。

练习题：看懂不算会，能判断结果才算稳

下面这些题适合读完对应章节后自测。不要急着展开答案，先说出现象，再说底层原因，最后给出修复方案。

flow {
    emit(1)
    emit(2)
}.map {
    println("map $it")
    it * 10
}.collect {
    println("collect $it")
}

输出顺序是 map 1、collect 10、map 2、collect 20。默认 Flow 是顺序管道，emit(1) 会等下游 map 和 collect 都处理完，才继续 emit(2)。

val job = viewModelScope.launch(Dispatchers.Default) {
    while (true) {
        calculateNext()
    }
}
job.cancel()

取消是协作式，不是抢占式。这个循环没有挂起点，也没有检查取消状态，所以可能持续运行。修复方式是在循环中使用 ensureActive()、yield() 或检查 isActive。

try {
    repository.sync()
} catch (t: Throwable) {
    report(t)
}

它会吞掉 CancellationException，破坏取消传播。正确写法是先捕获取消并重新抛出，再处理业务异常：catch (e: CancellationException) { throw e }。

flowOf(1)
    .catch { emit(0) }
    .collect {
        error("render failed")
    }

catch 只包住它上游的异常。collect 块属于下游，因此这里的异常不会被前面的 catch 捕获。要处理 collect 内部异常，应在 collect 外层使用 try/catch，或者把可能失败的逻辑移动到上游操作符中。

StateFlow 表达“当前状态”，会持有最新值并在新订阅者收集时立即发出。导航、Toast、Snackbar 属于一次性事件，重新订阅后重复消费会造成重复跳转或重复提示。更合适的是 SharedFlow(replay = 0) 或明确的事件通道。

// 不推荐
suspend fun load() = withContext(Dispatchers.IO) { api.load() }

// 推荐
class Repo(
    private val api: Api,
    private val io: CoroutineDispatcher
) {
    suspend fun load() = withContext(io) { api.load() }
}

生产代码注入 Dispatchers.IO，测试代码注入 StandardTestDispatcher 或 UnconfinedTestDispatcher。这样测试可以用虚拟时间和 advanceUntilIdle() 控制协程推进。

进阶：反模式诊断（你能找出 bug 吗？）

// Android 项目中的这段代码, 可能在哪里崩？
class MyViewModel : ViewModel() {
    fun loadUser() {
        viewModelScope.launch {
            val user = repository.getUser()
            _state.value = user         // ← 这里可能崩？
        }
    }
}
// repository.getUser() 是: suspend fun getUser() = api.fetchUser()

诊断：repository.getUser() 没有用 withContext(Dispatchers.IO) 包裹，在 Main 线程执行网络请求会 NetworkOnMainThreadException。viewModelScope 底层是 Dispatchers.Main，launch 启动的协程默认继承父的 Dispatcher = Main。解决方案：Repository 内部应该自己切到 IO，调用方不需要关心线程。

// Activity 中四个地方都在 collect 同一个 Flow
// 会发生什么？
class MyActivity : AppCompatActivity() {
    val flow = repository.userFlow()   // flow { emit(api.fetch()) }
    
    override fun onCreate() {
        lifecycleScope.launch { flow.collect { update1(it) } }
        lifecycleScope.launch { flow.collect { update2(it) } }
        lifecycleScope.launch { flow.collect { update3(it) } }
        lifecycleScope.launch { flow.collect { update4(it) } }
    }
}

诊断：每个 collect 都会触发一次完整的网络请求——一共 4 次！因为 userFlow() 是冷流，每个收集者独立。解决方案：用 stateIn(scope) 把冷流转成热流，或 shareIn(scope) 共享给多个收集者。

runBlocking {
    val job = launch(Dispatchers.Default) {
        try {
            delay(100)
            println("1")
        } finally {
            println("2")
        }
    }
    delay(10)
    job.cancelAndJoin()
    println("3")
}

输出：2、3。不会输出 1——cancel 在 delay(100) 完成之前就触发了，协程在 delay 这个挂起点检测到取消。finally 仍然执行（输出 2），然后协程终止。3 在 cancelAndJoin 之后输出。

coroutineScope {
    val d1 = async { throw RuntimeException("A") }
    val d2 = async { delay(100); println("B") }
    try { d1.await() } catch (e: Exception) {}
    d2.await()   // ← 这里会怎样？
}

d2.await() 会抛异常。虽然 d1 的异常被 try/catch 吃掉了，但 d1 的 Job 在抛异常时已经失败了。普通 coroutineScope 下，一个子 Job 失败会取消父 Job，父 Job 取消 d2——所以 d2 在被 cancel 后 await 会抛 CancellationException。如果改成 supervisorScope，d2 不受影响，能正常打印 "B"。

大厂面试题库：回答要有“现象 + 原理 + 实战边界”

下面这些题覆盖 Android 协程面试的大部分核心盘问。点击展开答案，建议自己先口述一遍，再对照补齐关键词。

编译器会把 suspend 函数改写成带 Continuation 参数的状态机。每个挂起点对应一个 label，挂起前保存局部变量和恢复位置，返回 COROUTINE_SUSPENDED；异步操作完成后通过 resumeWith 重新进入状态机。
阻塞会占住当前线程等待结果；挂起会保存协程状态并返回，当前线程可以执行其他任务。恢复时由 Continuation 和 Dispatcher 决定在哪里继续执行。
launch 返回 Job，表达“只关心完成和取消”；async 返回 Deferred，表达“未来会有结果”。launch 根协程未捕获异常会走 CEH；async 的异常会保存在 Deferred 中，await 时抛出。但在普通父子结构中，async 失败仍会默认取消父 Job。
CEH 只处理根协程未捕获异常。它不会处理子协程局部异常、不会替代 try/catch、不会处理 async 已保存到 Deferred 的异常，也不会处理 CancellationException。
普通 Job 中子协程失败会取消父协程，父再取消兄弟协程。SupervisorJob 隔离子失败，子异常不自动取消父和兄弟，适合页面多个模块互不影响的加载场景。
withContext 是挂起函数，会切换上下文并等待代码块完成，返回结果；launch 会启动一个新协程，立即返回 Job。需要顺序拿结果用 withContext，需要并发任务用 launch/async。
flow{} 的 block 只有 collect 时才执行；每次 collect 都重新执行一遍，收集者之间默认互不共享。所以它适合描述数据生产逻辑，而不是保存 UI 状态。
flowOn 只影响它上游的 Flow 及操作符，不影响下游 collect。它通常通过上下游协程和 channel 边界实现上下文切换，同时保持 Flow 的上下文规则。
catch 捕获的是上游异常，包括 emit 和上游操作符抛出的异常。collect 块里的异常属于下游，不会被前面的 catch 捕获，需要在 collect 外层 try/catch。
页面状态用 StateFlow，因为它有初始值、保留最新值、可同步读取 value、按 equals 去重。一次性事件用 SharedFlow，因为它可以不重放，避免旋转屏幕或重新订阅时重复消费。
它绑定 ViewModel 生命周期，ViewModel 清理时自动取消子协程；底层通常是 SupervisorJob + Main Dispatcher。这样页面任务不会泄漏到 ViewModel 之外。
它让 Flow 收集随着生命周期自动启动和取消。页面不可见时取消收集，重新可见时再次收集，避免 UI 不可见时仍刷新或泄漏。
GlobalScope 不绑定业务生命周期，任务失败和取消都难以管理，容易造成泄漏和后台任务失控。除非是进程级、明确生命周期的全局任务，否则应使用业务 Scope。
在 coroutineScope 中用 async 启动两个子任务，再 await 合并。这样外层会等待所有子任务；任何一个失败会取消另一个并把异常抛给调用方。如果希望部分失败可用，用 supervisorScope + runCatching。
CancellationException 是协程取消的正常信号。如果 catch Throwable 后吞掉它，父协程以为任务正常完成，取消传播被破坏，可能导致资源继续运行或状态错误。
使用 kotlinx-coroutines-test 的 runTest、TestDispatcher 和 TestScheduler。生产代码通过构造函数注入 Dispatcher，测试中替换为 StandardTestDispatcher，再用 advanceUntilIdle 或 advanceTimeBy 推进虚拟时间。
